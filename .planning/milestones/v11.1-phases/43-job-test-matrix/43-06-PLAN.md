<objective>
Fix a single-line exception handling defect in the `POST /jobs` route handler. The `HTTPException(422)` raised by `job_service.create_job()` is currently swallowed by a blanket `except Exception as e: raise HTTPException(status_code=500, ...)` clause because `HTTPException` is a subclass of `Exception`. The fix is to add `except HTTPException: raise` before the generic clause.

Purpose: Unblocks JOB-05's 422 rejection assertion. Without this fix, the live stack returns HTTP 500 with the 422 detail embedded as a string, which causes `verify_job_05_env_routing.py` to fail even with nodes online.

Output: `main.py` POST /jobs route returns HTTP 422 (not 500) when no eligible node matches the requested env_tag.
</objective>

```python
@app.post("/jobs", response_model=JobResponse, tags=["Jobs"])
async def create_job(job_req: JobCreate, current_user: User = Depends(require_auth), db: AsyncSession = Depends(get_db)):
    try:
        result = await JobService.create_job(job_req, db)
        await ws_manager.broadcast("job:created", {"guid": result["guid"], "status": "PENDING", "task_type": job_req.task_type})
        return result
    except HTTPException:
        raise
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))
```

This is the standard FastAPI pattern for letting intended HTTP errors propagate while still catching unexpected exceptions as 500.
