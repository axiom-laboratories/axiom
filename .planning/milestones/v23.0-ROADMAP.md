**Granularity:** Fine (6 phases for clean requirement grouping)

## Overview

v23.0 transforms Axiom from a single-job dispatch platform into a comprehensive workflow orchestration system. Users can compose ScheduledJob steps into named Workflow templates, trigger them manually, on cron schedules, or via webhooks with cryptographic validation, and track execution through a WorkflowRun entity with step-level status visibility and visual DAG representation.

**Key capabilities:**
- Workflow data model with DAG validation (cycle detection, depth limit 30)
- WorkflowRun execution engine with atomic concurrency guards (SELECT...FOR UPDATE) and cascade failure propagation
- Six node types: Script steps, IF conditionals, AND/JOIN synchronization, OR, parallel fan-out, signal wait
- Three trigger modes: manual (with runtime parameters), cron (APScheduler), webhook (HMAC-SHA256 + nonce replay protection)
- Parameter injection as WORKFLOW_PARAM_* environment variables into step containers
- Read-only DAG visualization with live status overlays and run history
- Visual drag-and-drop editor for Workflow composition

## Phase Breakdown

### Phase 146: Workflow Data Model

**Goal:** Users can define named Workflows as a DAG of ScheduledJob steps with validated edges, automatically detect cycles, and manage the definition lifecycle (create/read/update/delete).

**Depends on:** Nothing (foundation phase)

**Requirements:** WORKFLOW-01, WORKFLOW-02, WORKFLOW-03, WORKFLOW-04, WORKFLOW-05

**Success Criteria:**
1. User creates a Workflow via `POST /api/workflows` with name, steps (array of ScheduledJob IDs), and edges (array of {from_step_id, to_step_id}); system validates DAG acyclicity and depth ≤30, returns Workflow entity with UUID and created_at
2. User lists all Workflows via `GET /api/workflows` with pagination and filters; response includes step_count, trigger_config, last_run_status for each Workflow
3. User updates a Workflow definition via `PUT /api/workflows/{id}` (edges, steps); system re-validates DAG on save and returns HTTP 422 if cycle detected or depth exceeds 30
4. User deletes a Workflow via `DELETE /api/workflows/{id}`; system blocks deletion (HTTP 409) if any active WorkflowRun exists for that Workflow
5. When user executes "Save as New" on a scheduled Workflow, system auto-pauses the original APScheduler schedule to prevent ghost execution

**Plans:** 3 plans (Wave 0: schema+tests, Wave 1: ORM+service, Wave 2: API routes)

### Phase 147: WorkflowRun Execution Engine

**Goal:** The system automatically orchestrates WorkflowRun execution via topological BFS dispatch, using atomic database guards to prevent race conditions when concurrent steps complete, and enforces a status state machine (RUNNING/COMPLETED/PARTIAL/FAILED/CANCELLED) with cascade failure propagation.

**Depends on:** Phase 146 (Workflow data model must exist)

**Requirements:** ENGINE-01, ENGINE-02, ENGINE-03, ENGINE-04, ENGINE-05, ENGINE-06, ENGINE-07

**Success Criteria:**
1. System dispatches WorkflowRun steps in topological order (BFS) via background task; each step releases only after all its dependencies complete; observer can view live status via `/api/workflows/{id}/runs/{run_id}` showing RUNNING with active step count
2. System accepts workflow-instantiated jobs with depth up to 30 (overriding the 10-level single-job limit) when created with `workflow_run_id` context
3. System uses atomic `SELECT...FOR UPDATE` transaction locks when processing concurrent step completions to prevent duplicate dispatch; unit tests verify serialized access
4. WorkflowRun status machine enforces: RUNNING (initial) → COMPLETED (all steps done) / PARTIAL (failed steps absorbed) / FAILED (unrecoverable) / CANCELLED (user abort)
5. When a step fails, system immediately cascades FAILED status to all downstream PENDING steps; observer sees a "failed upstream" marker on cascade-cancelled steps
6. System marks WorkflowRun PARTIAL (not FAILED) when some downstream branches completed successfully despite failures in sibling branches (e.g. OR gate routes around failed branch)
7. User cancels a running WorkflowRun via `POST /api/workflows/{id}/runs/{run_id}/cancel`; system aborts all ASSIGNED/RUNNING step jobs and marks all PENDING steps CANCELLED; observer sees status update within 2 seconds

### Phase 148: Gate Node Types

**Goal:** Workflows support six specialized node types: IF gates for conditional routing, AND/JOIN for synchronization, OR for alternative paths, parallel fan-out for concurrent execution, and signal wait for inter-workflow coordination.

**Depends on:** Phase 147 (execution engine must exist to dispatch gates)

**Requirements:** GATE-01, GATE-02, GATE-03, GATE-04, GATE-05, GATE-06

**Success Criteria:**
1. IF gate evaluates a condition expression against the previous step's `/tmp/axiom/result.json` output using operators (eq, neq, gt, lt, contains, exists); returns first matching branch or marks step FAILED if no match; observer sees condition routing in run logs
2. When IF gate has no matching branch, system marks the step FAILED and cascades cancellation to all downstream PENDING steps
3. AND/JOIN gate holds PENDING until all incoming branches complete; then releases downstream steps; observer sees "waiting for N branches" status during hold
4. OR gate releases downstream steps when any single incoming branch completes; observer sees which branch triggered the OR in run history
5. Parallel fan-out node routes to multiple independent downstream branches concurrently; observer sees all branches execute in parallel in run visualization
6. Signal wait node pauses WorkflowRun step execution until named signal is posted via existing `/api/signals/{signal_name}/post`; observer sees step in WAITING status; posting signal unblocks it

### Phase 149: Triggers & Parameter Injection

**Goal:** Workflows can be triggered manually from the dashboard with runtime parameters, scheduled on cron expressions, or invoked via webhook with HMAC-SHA256 validation and nonce replay protection; all parameter values are injected as WORKFLOW_PARAM_* environment variables into step containers.

**Depends on:** Phase 146 (Workflow model), Phase 147 (execution engine to dispatch triggered runs)

**Requirements:** TRIGGER-01, TRIGGER-02, TRIGGER-03, TRIGGER-04, TRIGGER-05, PARAMS-01, PARAMS-02

**Success Criteria:**
1. User manually triggers a WorkflowRun via `POST /api/workflows/{id}/trigger` with a JSON object of parameter values; system validates parameter types against Workflow definition and dispatches WorkflowRun immediately; observer sees new run in history
2. User schedules a Workflow on a cron expression via `POST /api/workflows/{id}/schedule` with cron string; system registers APScheduler trigger and auto-pauses it on "Save as New" clone to prevent ghost execution; observer sees next-run time in dashboard schedule view
3. User configures webhook endpoint `POST /api/webhooks/{workflow_id}/trigger` via dashboard; system generates unique webhook URL and HMAC secret
4. Webhook endpoint validates all three security headers: X-Webhook-Signature (HMAC-SHA256 of body), X-Webhook-Timestamp (within ±5 min), X-Webhook-Nonce (checked against 24h dedup cache); rejects HTTP 400 and audit-logs all rejections (bad signature, stale timestamp, replayed nonce)
5. User defines named parameters on Workflow definition (name, type: string|int|bool, optional default); system injects each runtime value as WORKFLOW_PARAM_<NAME>=<value> environment variable into every step's container; signed script content is never modified
6. System stores parameter values in WorkflowRun entity for audit trail; observer can view parameter values in run detail view

### Phase 150: Dashboard Read-Only Views

**Goal:** Users visualize Workflow definitions and executions via read-only DAG diagrams with live status overlays, inspect run history, drill into step logs and structured output, and see Workflows and ScheduledJobs unified in a single schedule view.

**Depends on:** Phase 146 (Workflow model), Phase 147 (execution engine for live runs), Phase 149 (triggers for scheduling context)

**Requirements:** UI-01, UI-02, UI-03, UI-04, UI-05

**Success Criteria:**
1. User views read-only DAG visualization of a Workflow on Workflows.tsx detail view; elkjs auto-layout renders step graph with proper layering and spacing; observer sees clear step dependencies and node types (script, IF, AND/JOIN, OR, parallel, signal)
2. During active WorkflowRun execution, live status overlay colors nodes by status (PENDING=grey, RUNNING=blue, COMPLETED=green, FAILED=red, CANCELLED=strikethrough); observer sees status change within 1 second of server update via WebSocket
3. User views run history for a Workflow via `GET /api/workflows/{id}/runs` with pagination; response includes trigger_type (manual/cron/webhook), status, started_at, completed_at, duration_seconds for each run
4. User drills into a WorkflowRunStep detail view; observer sees job output (stdout/stderr), live logs during execution, and `/tmp/axiom/result.json` structured output (JSON pretty-printed)
5. Unified schedule view combines ScheduledJob (badge: "JOB", shows countdown to next run) and Workflow (badge: "FLOW", shows next cron fire if scheduled) entries in a single sortable/filterable list with last-run status and next-run time

### Phase 151: Visual DAG Editor

**Goal:** Users compose Workflows visually by dragging ScheduledJob steps onto an interactive canvas, connecting them with directed edges, configuring IF gate conditions inline, and receiving real-time validation feedback (cycle warnings, depth approaching 30, missing condition branches).

**Depends on:** Phase 146 (Workflow model), Phase 148 (gate node types for IF config), Phase 150 (read-only DAG view as reference)

**Requirements:** UI-06, UI-07

**Success Criteria:**
1. User opens Workflow editor (new Workflow or existing); canvas displays current DAG (empty for new); user drags ScheduledJob steps from sidebar onto canvas, each becomes a node; user draws directed edges between nodes to define dependencies; real-time validation highlights cycles in red ("cycle detected") and enforces depth limit (amber warning if depth > 25, red at >30)
2. User configures IF gate node inline: clicks gate node → popover shows condition editor with fields for (operator, field_from_result_json, expected_value); observer sees condition text rendered on gate node; dragging edge from IF gate to step marks edge as "branch_name" in data model
3. System prevents invalid graphs: blocks edge creation that would create cycle (red cursor feedback); warns on unmatched IF branches (amber "unmatched condition"); blocks save if depth >30

## Architecture Notes

**Stack:**
- Core engine: FastAPI (async, existing) + React (existing)
- Cycle detection: `networkx` library (add to `puppeteer/requirements.txt` in Phase 146)
- Visual editor dependencies (Phase 151 only): `@xyflow/react` v12.8.3, `elkjs` v0.11.1, `zustand` v5.4.0
- DAG layout: elkjs for auto-layout (phase-based hierarchical); ReactFlow for interactive canvas

**DB Tables (new):**
- `workflows` (id, name, created_by, created_at, updated_at, is_paused) — NO definition_json blob
- `workflow_steps` (id, workflow_id, scheduled_job_id, node_type, config_json)
- `workflow_edges` (id, workflow_id, from_step_id, to_step_id, branch_name)
- `workflow_runs` (id, workflow_id, status, started_at, completed_at, trigger_type, triggered_by)
- `workflow_run_steps` (id, workflow_run_id, workflow_step_id, status, job_id, result_json)
- `workflow_parameters` (id, workflow_id, name, type, default_value)
- `webhook_nonces` (id, workflow_id, nonce_hash, expires_at)

**Concurrency safeguards:**
- Phase 147 (ENGINE-03) is critical — `SELECT...FOR UPDATE` transaction locks prevent race conditions when multiple steps complete concurrently
- Background task (100ms interval) with idempotent unblock guard (`last_unblock_attempt_at`)
- Atomic step completion → result validation → downstream dispatch

**Security:**
- PARAMS-02: Environment variable injection only — signed script content never modified
- TRIGGER-04..05: HMAC-SHA256 validation, ±5 min timestamp, 24h nonce dedup
- Result.json validation: JSON schema check before IF gate consumption; optional signature verification

**Backward compatibility:**
- Coexists with existing ScheduledJob + job execution paths
- New workflow_run_id context field on jobs (nullable, preserves existing job schema)
- Depth limit override (30 vs 10) only applies to workflow-context jobs
