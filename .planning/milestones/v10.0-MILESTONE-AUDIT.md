## Integration Findings

### Missing Connection — MISSING-01 (Critical)

**`max_retries` absent from `ExecutionRecordResponse`**

- **From:** Phase 29 — `Job.max_retries` DB column exists; `WorkResponse.max_retries` Pydantic field exists
- **To:** Phase 32 — `DefinitionHistoryPanel` (in `JobDefinitions.tsx`) reads `row.max_retries` to render retry badge
- **Bug:** `ExecutionRecordResponse` (models.py ~line 624) has `attempt_number` and `job_run_id` but no `max_retries`. `list_executions()` does not join `max_retries` from the `Job` table.
- **Runtime effect:** Badge condition `(row.max_retries ?? 0) > 1` always false → retry badge permanently hidden, even on jobs with `max_retries=3` that have failed multiple times
- **Tests not catching it:** Test suite does not mock `max_retries` in execution response data → passes green while badge is broken
- **Fix:** Add `max_retries: Optional[int] = None` to `ExecutionRecordResponse`; join from `Job` in `list_executions()` query

### Info — INFO-01

**`attestation_verified` edge case when `job.node_id` is None**

- `attestation_service.verify_bundle()` called with `node_id=job.node_id`. If `job.node_id` is None at result-report time, returns `ATTESTATION_FAILED` instead of `ATTESTATION_MISSING`.
- Edge case only — happy path unaffected. Acceptable as known tech debt.
