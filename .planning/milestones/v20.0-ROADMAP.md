# Roadmap: v20.0 Node Capacity & Isolation Validation

**Milestone:** Node Capacity & Isolation Validation
**Defined:** 2026-04-06
**Phase Range:** 120–128 (9 phases)
**Granularity:** fine

**Core Value:** Prove that resource limits set via the GUI are enforced end-to-end through nested containers, that all jobs execute in ephemeral containers (never directly on the node), and that concurrent jobs are isolated from each other.

---

## Phase Details

### Phase 120: Database & API Contract
**Goal**: Establish database schema and API models for memory/CPU limits so downstream layers can persist and query them
**Depends on**: Nothing (foundational)
**Requirements**: ENFC-03 (limits reach end-to-end, starting with DB)
**Success Criteria** (what must be TRUE):
  1. Job table has memory_limit and cpu_limit columns (nullable, stored as strings for flexibility)
  2. Migration_v14.sql handles both fresh deployments and existing Postgres instances (IF NOT EXISTS)
  3. JobCreate, JobResponse, and WorkResponse Pydantic models expose memory_limit and cpu_limit fields
  4. All three response models serialize limits correctly in API responses (GET /jobs, POST /dispatch, GET /work/pull)

### Phase 121: Job Service & Admission Control
**Goal**: Implement API-level validation and persistence so oversized jobs are rejected before reaching nodes
**Depends on**: Phase 120
**Requirements**: ENFC-03 (limits reach API), ENFC-01 (memory enforcement foundation)
**Success Criteria** (what must be TRUE):
  1. job_service.create_job() stores memory_limit and cpu_limit to DB when provided
  2. parse_bytes() helper validates memory format (512m, 1g, 1Gi, etc.) and rejects invalid inputs
  3. job_service.select_best_node() performs admission check: job.memory_limit vs node.job_memory_limit (reject if oversized)
  4. API returns 422 (Unprocessable Entity) if memory_limit format is invalid or job exceeds node capacity

**Plans**: TBD

### Phase 122: Node-Side Limit Integration
**Goal**: Node extracts limits from work queue and passes them to the runtime engine for container enforcement
**Depends on**: Phase 120, Phase 121
**Requirements**: ENFC-03 (limits reach inner container runtime), EPHR-01 (foundation for isolated execution)
**Success Criteria** (what must be TRUE):
  1. node.py execute_task() extracts memory_limit and cpu_limit from work dict
  2. Secondary admission check in node.py rejects jobs that exceed node's available capacity
  3. Extracted limits passed to runtime.run(memory_limit=..., cpu_limit=...) for container runtime
  4. Node logs parse errors and fails job with diagnostic message if limit format invalid at node level

**Plans**: TBD

### Phase 123: Cgroup Detection Backend
**Goal**: Node detects cgroup v1 vs v2 vs unsupported at startup and reports in heartbeat
**Depends on**: Nothing (parallel with DB phases)
**Requirements**: CGRP-01 (startup detection), CGRP-02 (heartbeat reporting)
**Success Criteria** (what must be TRUE):
  1. CgroupDetector class detects cgroup version from /proc/self/cgroup and /sys/fs/cgroup/cgroup.controllers
  2. Node startup logs detected cgroup version (v1 / v2 / unsupported)
  3. Heartbeat payload includes detected_cgroup_version field sent to orchestrator every N seconds
  4. Orchestrator stores cgroup_version on Node table for dashboard + operator visibility

**Plans**: TBD

### Phase 124: Ephemeral Execution Guarantee
**Goal**: All job code executes inside ephemeral containers; EXECUTION_MODE=direct flagged and blocked in production
**Depends on**: Phase 122 (limits already require container isolation)
**Requirements**: EPHR-01 (no direct execution), EPHR-02 (EXECUTION_MODE=direct safety)
**Success Criteria** (what must be TRUE):
  1. runtime.py raises RuntimeError if EXECUTION_MODE=direct is active (no subprocess fallback)
  2. Node startup logs amber warning if EXECUTION_MODE=direct detected; admin.tsx shows UNSAFE badge
  3. Node rejects work-pull requests if EXECUTION_MODE=direct + production environment detected
  4. All existing jobs verified to execute via ephemeral containers, never directly on node host

**Plans**: TBD

### Phase 125: Stress Test Corpus
**Goal**: Comprehensive stress test suite (Python, Bash, PowerShell) for memory, CPU, and noisy-neighbour validation
**Depends on**: Phase 120, Phase 121 (API contract stable)
**Requirements**: STRS-01 (CPU burner), STRS-02 (memory hog), STRS-03 (noisy-neighbour monitor), STRS-04 (pre-flight check), STRS-05 (test orchestrator)
**Success Criteria** (what must be TRUE):
  1. Five test scripts in mop_validation/scripts/: cpu_burner.py/sh/.ps1, memory_hog.py/sh/.ps1, control_monitor.py/sh/.ps1 (all 3 languages)
  2. Pre-flight cgroup check validates node environment (cgroup version, runtime availability, resource availability)
  3. stress_orchestrator.py dispatches jobs via REST API, collects metrics (memory/CPU/latency), validates enforcement
  4. All scripts log structured output (JSON or key=value) for metrics extraction and automated pass/fail determination

**Plans**: TBD

### Phase 126: Limit Enforcement Validation
**Goal**: Validate CPU limits and test on both Docker and Podman job execution runtimes
**Depends on**: Phase 122 (node limit integration), Phase 125 (stress corpus for testing)
**Requirements**: ENFC-01 (memory OOMKill), ENFC-02 (CPU limit caps cores), ENFC-04 (Docker/Podman validation)
**Success Criteria** (what must be TRUE):
  1. Memory limit triggers OOMKill (exit code 137) when exceeded; stress corpus confirms via automated tests
  2. CPU limit (e.g., "2") caps available cores; stress corpus measures and validates no more than N cores used
  3. Both Docker and Podman runtimes enforce limits correctly (tested with stress corpus on dual-runtime node)
  4. Integration tests pass on both cgroup v1 and v2 systems (CI pipeline validates)

**Plans**: TBD

### Phase 127: Cgroup Dashboard & Monitoring
**Goal**: Dashboard shows cgroup version badge and operator warnings for degraded cgroup support
**Depends on**: Phase 123 (cgroup detection backend stable)
**Requirements**: CGRP-03 (dashboard badge), CGRP-04 (operator warning)
**Success Criteria** (what must be TRUE):
  1. Nodes.tsx shows cgroup version badge per node (v1 / v2 / unsupported) with color coding
  2. Amber warning badge displayed for v1 or unsupported cgroup nodes
  3. Tooltip or detail pane explains cgroup version impact (v2 recommended, v1 limits supported, unsupported = no limits)
  4. Admin.tsx system configuration page shows cgroup compatibility summary across fleet

**Plans**: TBD

### Phase 128: Concurrent Isolation Verification
**Goal**: Verify two concurrent jobs on same node remain isolated; latency drift stays below threshold
**Depends on**: Phase 125 (stress corpus), Phase 126 (enforcement validated)
**Requirements**: ISOL-01 (memory hog doesn't starve neighbour), ISOL-02 (control monitor detects <1.1s sleep drift)
**Success Criteria** (what must be TRUE):
  1. Automated test dispatches memory-hog job (60% node RAM) + compute job concurrently; both complete successfully
  2. Control monitor detects sleep latency drift <1.1s (confirms noisy-neighbour protection works)
  3. Neither job timeout or fail; output shows isolation maintained (memory RSS stayed within assigned limit)
  4. Dashboard execution history shows both jobs marked as COMPLETED, no cross-contamination in resource usage
