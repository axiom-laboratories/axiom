## Critical Integration Gaps

### INT-01 — CRITICAL: Script Key Mismatch

**`script` vs `script_content` in GuidedDispatchCard**

- **From:** `GuidedDispatchCard.tsx:160` — `payload: { script: form.scriptContent }`
- **To:** `node.py:553` — `payload.get("script_content")`

Every job submitted via the guided form arrives at the node with an empty script. The node calls `security_rejected=True` with error "Missing script or signature". This blocks the primary new operator UX of the entire milestone.

The HMAC computation in `job_service.py:1088` also uses `script_content`, computing HMAC on an empty string — so SEC-02 triggers a false positive HMAC mismatch for all guided-form jobs.

**Affected:** JOB-01, RT-01, RT-02

**Fix (1 line):** Change `GuidedDispatchCard.tsx:160` from `payload: { script: ... }` to `payload: { script_content: ... }`.

---

### INT-02 — HIGH: Queue View Always Empty

**Double `/api/` prefix in Queue.tsx**

- **From:** `Queue.tsx:113,124` — `authenticatedFetch('/api/jobs')`, `authenticatedFetch('/api/nodes')`
- **To:** FastAPI routes at `/jobs` and `/nodes` (after Caddy strips the `/api/` prefix)
- **Result:** Final URLs become `/api/api/jobs` → Caddy strips one `/api/` → FastAPI receives `/api/jobs` → 404 (no such route)

All other views use `authenticatedFetch('/jobs')` correctly. Queue.tsx is uniquely broken.

**Affected:** VIS-02

**Fix (2 lines):** Change Queue.tsx to `authenticatedFetch('/jobs')` and `authenticatedFetch('/nodes')`.

### INT-03 — HIGH: Execution CSV Export 404

**Route mismatch between Jobs.tsx and main.py**

- **From:** `Jobs.tsx:262` — `authenticatedFetch('/jobs/${job.guid}/executions/export')`
- **Backend:** `main.py:2357` — `@app.get("/api/jobs/{guid}/executions/export")`
- After Caddy strips `/api/`: FastAPI expects `/jobs/{guid}/executions/export` ✓ — this should work.

Wait — re-examining: the commit `c5a6819` changed Phase 53 routes to add `/api/` prefix on the *frontend* calls. If the export call at line 262 was also updated to `/api/jobs/{guid}/executions/export`, it would double-prefix. If it *wasn't* updated, it's consistent with the backend. The integration checker flagged this as broken — the exact root cause depends on which side of the c5a6819 fix the export call landed on.

**Affected:** SRCH-10

**Fix:** Confirm whether Jobs.tsx export call has `/api/` prefix or not, then ensure it matches the backend route registration pattern.

### INT-04 — MEDIUM: Retry State + Provenance Missing from Job List

**list_jobs response dict omits retry and originating fields**

`job_service.py` `list_jobs()` (lines 183-199) does not include `retry_count`, `max_retries`, `retry_after`, or `originating_guid` in the returned dict. These fields exist on the Job ORM model and are declared `Optional` on `JobResponse`, so they serialize as absent (null).

The `JobDetailPanel` retry countdown and attempt counter depend on these fields and never render. The "Resubmitted from" provenance link also never appears, making JOB-05 traceability invisible to operators.

**Affected:** JOB-04 (partial), JOB-05 (partial)

**Fix:** Add the four fields to the `list_jobs()` response dict mapping, or implement `GET /jobs/{guid}` single-job endpoint.
