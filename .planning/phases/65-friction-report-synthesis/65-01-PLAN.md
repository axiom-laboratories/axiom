**Parsing (use the verified regex patterns from the context interfaces block):**
- Extract `### [Category] Title` blocks using BLOCK_RE (MULTILINE + DOTALL, `\Z` in lookahead for last block)
- Per block: extract heading category+title, Classification line, What happened line, presence of Fix/Fix applied
- Severity tier from Classification: check for ROUGH EDGE before ROUGH (two-word tier); tiers: BLOCKER, NOTABLE, ROUGH EDGE, MINOR
- Harness-only detection on Classification text: keywords "automated harness", "HOME isolation", "agent run", "evaluation harness" → status = Harness-only
- Fixed-during-run detection: FIX_RE match in block body → status = Fixed during run (independent of Classification text)
- Edition attribution per file: CE-INSTALL/CE-OPERATOR → CE, EE-INSTALL/EE-OPERATOR → EE

**Deduplication:**
- After collecting all findings from all 4 files, group by normalised title: lowercase, strip `[Category]` prefix
- If normalised title appears in >=2 files from different editions → merge to single Finding, attribution = Shared
- Keep first-seen text as canonical; union the editions

**Report sections (write in this order):**
1. Executive Summary: run metadata (date, editions), finding counts by severity (BLOCKER/NOTABLE/ROUGH EDGE/MINOR), count of harness-only excluded, verdict line
2. Cross-Edition Comparison Table: markdown pipe table, columns: `Finding | Severity | CE | EE | Status | Fix Target`
   - CE/EE columns: check-mark if finding present in that edition, dash otherwise
   - Status: Open / Fixed during run / Harness-only
   - Fix Target: actionable file path (e.g., `docs/getting-started/install.md`)
   - Shared findings get check-mark in both CE and EE columns
3. Findings by severity: three sections — BLOCKER, NOTABLE, ROUGH EDGE (in that order)
   - Each finding: `#### Title`, what happened (1-2 sentences from What happened field), editions affected, status, and "**Recommendation:**" with specific file path + one-sentence description of what to change
   - BLOCKER and NOTABLE MUST have a Recommendation line; ROUGH EDGE recommended but can be brief
   - Harness-only findings included in the BLOCKER section but labelled "(Harness-only — excluded from verdict)"
4. First-User Readiness Verdict: binary NOT READY / READY
   - NOT READY if any BLOCKER where status != Harness-only (Fixed during run counts as open per user decision)
   - List each blocking criterion by title

**Fix Target guidance (hard-code these based on the pre-analysed findings in RESEARCH.md):**
