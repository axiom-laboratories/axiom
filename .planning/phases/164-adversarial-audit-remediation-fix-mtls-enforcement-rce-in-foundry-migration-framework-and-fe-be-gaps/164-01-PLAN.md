  <files>puppeteer/agent_service/security.py</files>
  <action>
Add `verify_client_cert()` function after the existing `verify_node_secret()` function. Implementation must:

1. Extract `X-SSL-Client-CN` header from the request (set by Caddy after successful TLS handshake)
2. Reject with 403 if header is missing or malformed
3. Parse node ID from CN (expected format: "node-{node_id}")
4. Look up the node in the `Node` table via `db.execute(select(Node).where(Node.id == node_id))`
5. Extract the certificate serial number from `node.client_cert_pem` using `cryptography.x509.load_pem_x509_certificate()`
6. Cross-check against `RevokedCert` table — reject with 403 if serial is found (defense-in-depth)
7. Return node_id (str) for downstream handlers to use

Error cases:
- Missing `X-SSL-Client-CN` header → HTTP 403 "Missing client certificate"
- Invalid CN format (not starting with "node-") → HTTP 403 "Invalid certificate CN format"
- Node not found in DB → HTTP 403 "Node not found"
- Certificate serial is in RevokedCert table → HTTP 403 "Certificate revoked"

Use the same async/Depends() pattern as `verify_node_secret()`. Import `Request` from fastapi, `select` from sqlalchemy, `AsyncSession` from sqlalchemy.ext.asyncio, and certificate parsing from `cryptography.hazmat.backends.default_backend()`.

Pattern reference: The existing `verify_node_secret()` function shows the Depends() pattern.

Make two changes to the Caddyfile:

**Change 1 — mTLS Client Auth Policy (SEC-01):**
Add a `client_auth` block that defines the mTLS policy for node-facing routes. Caddy should:
- Require client certificates on routes matching `@mtls_clients` (which will be `/work/pull` and `/heartbeat`)
- Use the CRL (Certificate Revocation List) served by the agent at `/system/crl.pem`
- Forward the client certificate CN via `X-SSL-Client-CN` header to the application
- Reject connections without valid client certs

Reference Caddy 2.x syntax:
```caddyfile
@mtls_clients {
    path /work/pull /heartbeat
}

(mtls_policy) {
    client_auth {
        mode require_and_verify
        trusted_ca_cert_file /etc/certs/internal-ca.crt
        # CRL validation: file-based or URL-based
        # option 1 (file-based): crl_file /var/certs/crl.pem
        # option 2 (URL-based): ask your architecture which approach
    }
    # Forward CN header to application:
    header X-SSL-Client-CN {tls_client_subject}
}
```

Apply this policy to the node-facing routes in the existing reverse_proxy block (or create a new block for mtls_clients).

**Note:** `/api/enroll` should NOT require client certificates (nodes don't have certs yet during enrollment). Exclude it from the `@mtls_clients` matcher if it's currently catching all requests to the agent.
