**Starter Template type** (from Wave 2):
```typescript
interface Template {
  id: string;
  friendly_name: string;
  is_starter: boolean;
  status: "ACTIVE" | "DEPRECATED" | "REVOKED";
  base_image?: string;
  package_count?: number;
  description?: string;
  blueprint?: RuntimeBlueprint;
}

interface RuntimeBlueprint {
  template_id: string;
  packages: {
    name: string;
    ecosystem: string;
    version_constraint?: string;
  }[];
}
```

**Existing build endpoint** (to be integrated):
- POST /api/templates/{id}/build → builds template, returns {build_id, status}

**New endpoints to create:**
- POST /api/templates/{id}/clone → clones starter, returns {id, friendly_name, is_starter: false}
- POST /api/templates/{id}/build (enhanced) → auto-approves packages + builds

  <action>
    Add new endpoint to foundry_router.py (or existing route handler):

    **POST /api/templates/{id}/clone**
    - Input: { friendly_name: str (optional) }
    - Output: PuppetTemplateResponse (cloned template)
    - Permission: foundry:write
    - Logic:
      a. Query source template by id
      b. Check is_starter is True (can only clone starters)
      c. Create new PuppetTemplate:
         - id = uuid()
         - friendly_name = input.friendly_name or "{original} (Custom)"
         - is_starter = False (cloned templates are custom, not starters)
         - status = "ACTIVE"
         - Copy blueprint definition from source
      d. Create new RuntimeBlueprint (copy from source)
      e. audit(db, current_user, "template:cloned", f"{source.friendly_name} → {new.friendly_name}")
      f. Return cloned template details

    - Error handling:
      - 404 if source template not found
      - 400 if is_starter is False (can't clone custom templates)
      - 403 if user lacks foundry:write

    Why: Clone pattern allows reuse while keeping starters immutable. New template is editable (is_starter=false).
  </action>

  <files>puppeteer/agent_service/ee/routers/foundry_router.py</files>
  <action>
    Modify existing build_template endpoint (or create if missing):

    **POST /api/templates/{id}/build**
    - Input: { auto_approve: bool (optional, default=true for starters) }
    - Logic enhancement:
      a. Query template by id
      b. If template.is_starter == True AND auto_approve == True:
         - Loop through template.blueprint.packages
         - For each package, call SmelterService.add_ingredient(db, ApprovedIngredientCreate(...))
         - This auto-triggers resolver + mirror for each
         - If ingredient already approved, skip (idempotent)
         - Log: "Auto-approved {count} packages for {template.friendly_name}"
      c. Proceed with standard build process (foundry_service.build_template)
      d. Return build result {build_id, status, build_logs}

    - Error handling:
      - If auto-approve fails for any package, log error but continue (best-effort)
      - Return 202 Accepted (async) with build queued

    Why: Removes manual approval step for non-developers. Resolver/mirror work happens automatically.
