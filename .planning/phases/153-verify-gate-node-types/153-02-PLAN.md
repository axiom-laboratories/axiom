## Gate Dispatch Logic (from RESEARCH.md)

**GATE-03 (AND_JOIN):** Lines 529-563 in workflow_service.py
- Waits for all predecessors to reach COMPLETED status
- When all complete, marks AND_JOIN step COMPLETED via CAS guard
- Continues to next wave dispatch (no job creation for gate nodes)

**GATE-04 (OR_GATE):** Lines 565-589 in workflow_service.py
- Waits for any single predecessor to reach COMPLETED status
- When any completes, marks OR_GATE COMPLETED via CAS guard
- Calls _mark_branch_skipped() on non-triggered branch predecessors
- Unblocks downstream steps

**GATE-05 (PARALLEL):** Lines 518-527 in workflow_service.py
- Marks COMPLETED immediately via CAS guard
- No job creation; next wave dispatch naturally fans out to all downstream branches

**CAS Guard Pattern:** UPDATE WHERE status='PENDING' followed by rowcount==1 check
- Prevents duplicate dispatch if multiple requests process same step concurrently
- Critical for correct semantics when dispatch_next_wave() is called multiple times
</context>
