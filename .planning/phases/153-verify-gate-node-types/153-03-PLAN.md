## SIGNAL_WAIT Logic (from RESEARCH.md lines 188-214)

**Phase 1 — Block:**
- dispatch_next_wave() encounters SIGNAL_WAIT step
- Marks RUNNING (no job created)
- Awaits signal via advance_signal_wait()

**Phase 2 — Wakeup:**
1. User posts signal via POST /api/signals/{name}
2. Signal creation endpoint calls workflow_service.advance_signal_wait(signal_name)
3. Wakeup logic:
   - Find all RUNNING WorkflowStepRun records
   - Check step's config_json for signal_name field
   - If matching, mark COMPLETED
   - Call advance_workflow() to dispatch next wave

**Critical:** Cancellation guard (Pitfall 4, lines 291-301)
- Before advancing SIGNAL_WAIT, check parent WorkflowRun status
- If parent is CANCELLED, do NOT wake up step
- Prevents zombie steps from running in cancelled runs
