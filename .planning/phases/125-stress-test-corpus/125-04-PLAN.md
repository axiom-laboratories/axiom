## Preflight Check Requirements

Runs as a dispatched job (not local). Validates:
1. Cgroup version (v1 vs v2 vs unsupported)
2. CPU controller enabled (memory.limit_in_bytes or memory.max exists)
3. Memory controller enabled (cpuset.cpus or cpu.max exists)
4. Own memory limit applied (read from /sys/fs/cgroup, compare to container memory_limit)

Output: JSON with per-check pass/fail breakdown.
Exit: 0 = all checks pass, 1 = at least one check fails

## Orchestrator Scenario Dispatch

**Scenario 1: Single CPU burn**
- Dispatch cpu_burn (Python) with cpu_limit=0.5, 10s timeout
- Expect: ratio < 0.8 (throttling detected) → PASS
- Expect: ratio >= 0.8 (no throttling) → INFO (not a failure; node may have spare capacity)

**Scenario 2: Single memory OOM**
- Dispatch memory_hog (Python) with memory_limit=128M, allocation=256M, 35s timeout
- Expect: exit code 137 (OOM-killed) → PASS
- Expect: exit code 2 (survived) → FAIL (limits not enforced)

**Scenario 3: Concurrent isolation**
- Simultaneously dispatch memory_hog + cpu_burn + noisy_monitor on same node
- memory_hog and cpu_burn have limits; monitor runs without limits
- Expect: noisy_monitor detects max_drift < 1.1s → PASS (isolation working)
- Expect: max_drift >= 1.1s → FAIL (noisy neighbour interference)

**Scenario 4: All-language sweep**
- For each language (python, bash, powsh) and each script type (cpu_burn, memory_hog, noisy_monitor):
  - Dispatch script with appropriate limits
  - Poll until completion
  - Parse JSON result
  - Aggregate results per language

Create orchestrator script that dispatches all stress test scenarios:

**High-level algorithm:**
1. Load secrets from `mop_validation/secrets.env` (ADMIN_PASSWORD, SERVER_URL)
2. Login to API and obtain JWT token
3. Get list of available nodes via `GET /nodes`
4. For each node:
   a. Dispatch preflight_check.py; parse JSON result
   b. If preflight fails, skip this node and try next
   c. If preflight passes, run all 4 scenarios on this node
5. Generate console table output and JSON report file

**Scenario dispatch pattern:**
- Load script from disk: `with open(f"stress/{lang}/{script_name}") as f: content = f.read()`
- Sign content: `sign_content(content, private_key)`
- Dispatch: `POST /dispatch` with JobCreate model (script_content, signature, memory_limit, cpu_limit)
- Poll: `GET /jobs/{job_id}` until status in [completed, failed], with exponential backoff (max 2 sec/attempt, max 60 sec total)
- Parse result: JSON.parse(stdout) to extract pass/fail and details

**Scenario 1: Single CPU burn (python)**
- Script: stress/python/cpu_burn.py
- Limits: cpu_limit=0.5, memory_limit=512m
- Timeout: 10s
- Parse JSON result for pass/fail

**Scenario 2: Single memory OOM (python)**
- Script: stress/python/memory_hog.py with MEMORY_SIZE_MB=256
- Limits: memory_limit=128m, cpu_limit=1.0
- Timeout: 35s
- Check exit code 137 (OOM-killed) → PASS; exit code 2 → FAIL

**Scenario 3: Concurrent isolation (concurrent dispatch)**
- Simultaneously dispatch 3 jobs on same node:
  1. memory_hog (memory_limit=512m) as Job A
  2. cpu_burn (cpu_limit=1.0) as Job B
  3. noisy_monitor (no limits) as Job C
- Use `asyncio.gather()` to dispatch all 3 concurrently
- Poll all 3 until completion
- Parse Job C (noisy_monitor) for max_drift; expect < 1.1s → PASS

**Scenario 4: All-language sweep**
- For each language in [python, bash, pwsh]:
  - For each script in [cpu_burn, memory_hog, noisy_monitor]:
    - Dispatch with appropriate limits
    - Poll until completion
    - Record result (pass/fail)

**Report generation:**
- Console table: Print to stdout using tabulate or manual formatting
- JSON file: Write to `mop_validation/reports/stress_test_{timestamp}.json` with full results

**Error handling:**
- If node preflight fails, log and skip to next node
- If all nodes fail preflight, print error and exit 1
- If a single job fails, record as FAIL in report but continue with next job
- Connection errors: retry with exponential backoff (up to 3 retries per request)

**Key implementation patterns:**
- Use existing `run_signed_job.py` pattern for signing + dispatch
- Use asyncio for concurrent job dispatch (Scenario 3)
- Use `requests.Session()` with connection pooling for multiple API calls
