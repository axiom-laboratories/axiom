async def require_permission(permission: str):

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Per-process cache | Each worker process has own copy of _perm_cache; permission grant/revoke in one worker not visible to others |
| WebSocket event loop | Exception handling missing finally block; resource cleanup skipped on unexpected errors |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-171-13 | Elevation of Privilege | Multi-worker permission cache | mitigate | In-memory _perm_cache removed entirely. All permission checks now query DB directly. Single-worker deployment unaffected (slightly slower: one extra SELECT per request, negligible at current scale). Multi-worker deployments now safe: each request sees current permissions in DB, not stale cache copy from another worker. |
| T-171-14 | Denial of Service | WebSocket resource leak | mitigate | WebSocket handler wrapped in try/finally. `ws_manager.disconnect(ws)` now called on all exit paths (normal, WebSocketDisconnect, RuntimeError, CancelledError). Prevents connection accumulation over time, avoiding eventual socket exhaustion. |
| T-171-15 | Elevation of Privilege | Permission cache invalidation miss | mitigate | `_invalidate_perm_cache()` function removed; no invalidation logic to get out of sync. With DB-per-request approach, no invalidation needed — permission changes visible immediately on next request. Eliminates race condition where invalidation call sites missed (e.g., via REST API typo, code path skip). |

</threat_model>
