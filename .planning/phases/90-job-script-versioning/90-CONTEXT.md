# Phase 90: Job Script Versioning - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Every save to a job definition (script, schedule, targeting, metadata) is preserved as an immutable version record. Historical executions link to the exact definition version that was active at dispatch time. Operators can view the script that ran for any specific execution and compare versions side-by-side.

</domain>

<decisions>
## Implementation Decisions

### Version storage
- **One unified `job_definition_versions` table** — full snapshot of the entire definition on every save
- Columns: `id`, `job_def_id` (FK → ScheduledJob.id), `version_number` (int, per-definition sequential), `script_content`, `signature_id` (FK → Signature.id), `cron_expression`, `target_tags`, `target_node_id`, `runtime`, `max_retries`, `backoff_multiplier`, `created_at`, `created_by`, `change_summary`, `is_signed` (bool)
- `Job` table gets a new `definition_version_id` FK column (→ `job_definition_versions.id`) stamped at dispatch
- Rationale: two-table split creates coordination complexity when both script and schedule change in one save; SCD on ScheduledJob would require rewriting all existing queries; unified snapshot is simple, queryable, and the storage cost for repeated script content is negligible

### Version creation trigger
- **Every save creates a new version row** — any change to any field (script, schedule, tags, targeting, retry config) writes a new version
- Optional: skip if nothing actually changed (idempotent save guard) — compare snapshot before writing
- **Version 1 is created when the job definition is first created** — every definition always has at least one version
- `change_summary` field captures what changed (e.g. "script updated", "schedule changed to `0 * * * *`", "target tags updated")

### Draft/live versioning with signing
- Saving a script change writes a new version row with `is_signed = False` (DRAFT)
- The version becomes LIVE when re-signed (is_signed = True, signature_id populated)
- The definition continues dispatching against the **previous signed version** until re-signing completes
- Non-script changes (schedule, tags, targeting) are written as signed immediately (no script change = no re-signing required)

### Execution history display
- Version number surfaces in the **definition's history tab** (JobDefinitions.tsx already has this tab)
- History tab shows an **interleaved timeline**: both job execution rows AND definition change rows (e.g. "• v2 — Schedule changed — thomas — Mar 29")
- Execution rows display a compact **version badge inline** (e.g. "v3") next to the status or job name
- Definition change rows are visually distinct from execution rows (lighter colour, different icon) and are **not clickable** — informational only

### Script viewer
- "View script" accessible from **two entry points**:
  1. Job execution detail sheet (Jobs.tsx) — "View script (v3)" action
  2. Definition history tab (JobDefinitions.tsx) — clicking a version row or execution row
- Script displayed in a **read-only modal/sheet with syntax highlighting** (Python/Bash/PowerShell) and a copy button
- **Side-by-side diff on demand** — "Compare with previous" button in the script viewer opens a v(n-1) vs v(n) diff view

### Claude's Discretion
- Exact diff library to use for side-by-side comparison (react-diff-viewer or similar)
- Whether DRAFT versions are shown in the history tab (recommend: yes, visually marked as unsigned)
- Whether `definition_version_id` is backfilled on existing Job rows from `scheduled_job_id` → current script content (recommend: no, leave null for pre-phase-90 jobs)
- Exact column layout for the interleaved timeline rows
- Whether `change_summary` is auto-generated by comparing fields or manually set by the service

</decisions>

<specifics>
## Specific Ideas

- "View script (v3)" label in the job detail sheet makes it clear the version number is shown, not just "view script"
- Definition change rows in the interleaved timeline should be clearly subordinate visually — they're context, not primary records
- DRAFT badge on unsigned versions makes it obvious to operators that the previous version is still being dispatched

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `db.py: ScheduledJob` — existing job definition model; `script_content` is currently a mutable Text field that will become read-only after versioning (current version read from `job_definition_versions` instead)
- `db.py: Job` — needs new `definition_version_id` column (nullable, FK to `job_definition_versions.id`)
- `scheduler_service.py: update_job_definition()` (line 462) — the primary mutation point; currently writes directly to `job.script_content`. Phase 90 adds a version snapshot write here before (or instead of) mutating the ScheduledJob row
- `scheduler_service.py: create_job_definition()` (line ~415) — also writes script_content; needs to create version 1
- `scheduler_service.py: dispatch_scheduled_job()` (line ~220) — creates the Job row; needs to stamp `definition_version_id`
- `JobDefinitions.tsx: definition-history query` (line 48) — already queries per-definition execution history; this feeds the interleaved timeline
- `Jobs.tsx: scriptContent` (lines 94, 986, 1095) — already surfaces script content in job detail sheet; "View script" action extends this

### Established Patterns
- `Config` key/value pattern: `select(Config).where(Config.key == "key")` — not needed here but shows DB patterns
- `ScheduledJob` create_all pattern: new `job_definition_versions` table will be picked up automatically by `Base.metadata.create_all`
- `Job.definition_version_id` is a new nullable column on existing table — requires `ALTER TABLE jobs ADD COLUMN definition_version_id VARCHAR REFERENCES job_definition_versions(id)` migration for existing deployments
- Signature verification pattern: `SignatureService.verify_payload_signature(sig.public_key, signature, script_content)` — used in `update_job_definition()`; the `is_signed` flag on a version row reflects whether this check has passed

### Integration Points
- `scheduler_service.py`: Version row written inside `create_job_definition()` and `update_job_definition()` — single transaction with the ScheduledJob mutation
- `scheduler_service.py: dispatch_scheduled_job()`: Must query the current signed version ID before creating the Job row, then stamp `job.definition_version_id`
- `main.py`: New endpoints needed — `GET /jobs/definitions/{id}/versions` (list all versions), `GET /jobs/definitions/{id}/versions/{version_num}` (get specific version with script content)
- `JobDefinitions.tsx`: History tab refactored to merge execution rows + version change rows into a single interleaved timeline; version badge added to execution rows
- `Jobs.tsx`: Job detail sheet gets "View script (vN)" action that calls the version endpoint

</code_context>
