    **docs/docs/runbooks/node-validation.md** — per-job reference guide.

    Header: "# Node Validation Job Library". One-paragraph intro: what the corpus is, where jobs live, that they must be signed before dispatch.

    For each of the 7 jobs, write an H2 section:
    - **H2 name** matching the manifest job name (e.g., `## hello-bash`)
    - **What it tests:** one sentence
    - **Required capabilities:** small table (Capability | Value) — or "None" if empty
    - **Dispatch** code block:
      ```bash
      axiom-push job push \
        --script tools/example-jobs/bash/hello.sh \
        --key signing.key \
        --key-id <your-key-id>
      # Then publish from the Staging tab in the dashboard
      ```
    - **Expected PASS output** fenced block (short representative excerpt)
    - **Expected FAIL output** fenced block (what failure looks like — for hello-world jobs this is N/A; use a short note)

    Special notes per job:
    - validation-volume-mapping: note that operator must set `AXIOM_VOLUME_PATH` env var in job dispatch payload and configure a matching volume mount on the node. Include a minimal docker-compose snippet showing the volume and `environment: AXIOM_VOLUME_PATH=/mnt/axiom-data`.
    - validation-network-filter: note the node must be configured with `--network=none` (Docker flag) before dispatch. The script does NOT create isolation — it validates existing isolation. AXIOM_BLOCKED_HOST defaults to 8.8.8.8.
    - validation-memory-hog: note that "OOM FAILED" is the expected success outcome. The job is working correctly when it FAILS. Include the "inversion logic" explanation: configure the target node with `JOB_MEMORY_LIMIT=128m` (or equivalent compose override) before dispatching. Add `!!! warning "Expected FAILED status"` admonition block.
    - validation-cpu-spin: note ratio < 0.8 on a 0.5-CPU-limited node is the pass condition. The job completes (exit 0) and reports measurements. Configure node with `JOB_CPU_LIMIT=0.5` before dispatching.

    End with an H2 "## Resource Limit Node Setup" section explaining:
    1. How to configure a test node with a memory limit (docker-compose env vars `JOB_MEMORY_LIMIT=128m`)
    2. How to configure a CPU limit (`JOB_CPU_LIMIT=0.5`)
    3. The `resource_limits_supported` capability must be registered on the node — show the capability registration step in the dashboard
    4. LXC caveat: cgroup v2 resource limit enforcement is unreliable on LXC-hosted nodes; test on a native Docker node
