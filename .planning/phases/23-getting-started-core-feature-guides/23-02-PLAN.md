Required env vars for puppeteer (secrets.env / .env):
  SECRET_KEY=<random-64-char-hex>
  ENCRYPTION_KEY=<fernet-key>          # Generate: python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
  API_KEY=<arbitrary-string>           # WARNING: security.py calls sys.exit(1) at import time if missing
  ADMIN_PASSWORD=<initial-admin-pass>  # Seeds admin user on first start only

Required node env vars (node-compose.yaml):
  NODE_TAGS=general,linux
  JOB_IMAGE=docker.io/library/python:3.12-alpine
  AGENT_URL=https://puppeteer-agent-1:8001   # Or host.docker.internal:8001 for Docker Desktop
  JOIN_TOKEN=<enhanced-base64-token>          # MUST be copied from dashboard Nodes page (NOT raw API)
  ROOT_CA_PATH=/app/secrets/root_ca.crt
  EXECUTION_MODE=direct                       # Use 'direct' inside Docker to avoid cgroup v2 issues

JOIN_TOKEN format (critical pitfall):
  - Raw API (POST /api/enrollment-tokens) returns: {"token": "<hex-string>"}
  - Dashboard "Copy JOIN_TOKEN" button returns: base64({"t": "<hex>", "ca": "<PEM>"})
  - Node logs "Detected Enhanced Token. Bootstrapping Trust..." on success with enhanced format
  - ALWAYS copy from dashboard Nodes page, not from raw API response

First job signing key requirement:
  - Dashboard job creation requires a registered Ed25519 public key in Signatures view
  - Without a registered key, job creation fails with 422 signature validation error
  - Must register key BEFORE dispatching first job

Ed25519 key generation (no extra tools needed):
  openssl genpkey -algorithm ed25519 -out signing.key
  openssl pkey -in signing.key -pubout -out verification.key

Network connectivity options for node → puppeteer:
  - Docker Desktop (Mac/Windows): AGENT_URL=https://host.docker.internal:8001
  - Linux Docker: AGENT_URL=https://172.17.0.1:8001  (or: ip route | awk '/default/ {print $3}')
  - Same compose network: AGENT_URL=https://puppeteer-agent-1:8001

Admonition types available: note, tip, warning, danger, info, success
Blueprint packages format gotcha: Must use {"python": [...]} dict, NOT a plain list

**docs/docs/getting-started/enroll-node.md**

1. Title: "# Enroll a Node"
2. Intro: "Nodes self-enroll over mTLS — they generate a certificate signing request and get a signed client cert from the control plane. You provide a JOIN_TOKEN that embeds the Root CA so the node can establish trust automatically."
3. **Step 1: Generate an enrollment token**
   - In the dashboard, go to **Nodes → Generate Token**
   - Click **Copy JOIN_TOKEN** — this copies the enhanced token (base64-encoded JSON with the CA embedded)
   - `!!! warning "Use the dashboard Copy button"` admonition: the raw API endpoint `POST /api/enrollment-tokens` returns only the token hex string. The enhanced JOIN_TOKEN (which includes the Root CA for mTLS bootstrap) is only available via the dashboard copy button. Nodes will fail enrollment if given the raw hex string.
4. **Step 2: Configure node connectivity**
   - Table showing AGENT_URL pattern for three scenarios:
     | Scenario | AGENT_URL |
     |----------|-----------|
     | Docker Desktop (Mac/Windows) | `https://host.docker.internal:8001` |
     | Linux Docker (bridge network) | `https://172.17.0.1:8001` |
     | Same Docker Compose network | `https://puppeteer-agent-1:8001` |
   - Linux note: verify with `ip route | awk '/default/ {print $3}'` if 172.17.0.1 is not the correct gateway
5. **Step 3: Create node compose file** — show `node-compose.yaml` with all env vars (use the verified list from interfaces block). Include `EXECUTION_MODE=direct` with a `!!! tip` admonition explaining why (avoids cgroup v2 issues with nested container runtimes inside Docker).
6. **Step 4: Start the node**
   ```bash
   docker compose -f node-compose.yaml up -d
   ```
7. **Step 5: Verify enrollment**
   - Check node logs: `docker compose -f node-compose.yaml logs -f`
   - Expected log line: `Detected Enhanced Token. Bootstrapping Trust...` followed by `Node enrolled successfully`
   - In dashboard: go to **Nodes** — the node should appear with status **ONLINE** within 30 seconds
8. Navigation footer: "**Next:** [First Job →](first-job.md)"

**docs/docs/getting-started/first-job.md**

1. Title: "# Your First Job"
2. Intro: "Jobs are Python scripts signed with an Ed25519 private key. The node verifies the signature before executing. This guide walks through generating a signing key, registering the public key, and dispatching your first job via the dashboard."
3. **Step 1: Generate a signing keypair**
   ```bash
   openssl genpkey -algorithm ed25519 -out signing.key
   openssl pkey -in signing.key -pubout -out verification.key
   ```
   - `!!! warning "Keep your private key safe"` admonition: never commit `signing.key` to git. Add it to `.gitignore`. Use a secrets manager in production.
4. **Step 2: Register the public key in the dashboard**
   - Go to **Signatures** in the dashboard sidebar
   - Click **Add Signature Key**
   - Paste the contents of `verification.key` into the field
   - Give it a descriptive name (e.g., "dev-operator-key")
   - Save — note the **Key ID** displayed (you will need this when submitting jobs)
5. **Step 3: Write and sign a test script**
   - Create `hello.py`:
     ```python
     print("Hello from Master of Puppets!")
     import platform
     print(f"Running on {platform.node()} ({platform.system()})")
     ```
   - Sign it using the admin_signer tool or openssl:
     ```bash
     # Using openssl
     openssl pkeyutl -sign -inkey signing.key -out hello.py.sig -rawin -in hello.py
     # Base64-encode the signature for the API
     base64 -w0 hello.py.sig > hello.py.sig.b64
     ```
   - `!!! tip "mop-push automates this"` admonition: the `mop-push` CLI handles signing and submission in one command. See the [mop-push CLI guide](../feature-guides/mop-push.md) for details. This manual method is shown here to make the mechanics visible.
6. **Step 4: Dispatch the job via the dashboard**
   - Go to **Jobs** in the dashboard sidebar
   - Click **New Job**
   - Fill in:
     - **Script**: paste the contents of `hello.py`
     - **Signature**: paste the base64 signature from `hello.py.sig.b64`
     - **Signature Key**: select the key registered in Step 2
     - **Target tags**: leave blank to target any available node, or enter `general` to match the default node tag
   - Click **Dispatch**
7. **Step 5: Verify the result**
   - The job appears in the Jobs list with status **PENDING**, then **ASSIGNED**, then **COMPLETED**
   - Click the job to expand the result — the output should show:
     ```
     Hello from Master of Puppets!
     Running on <node-hostname> (Linux)
     ```
   - `!!! success "You've completed the Getting Started guide"` admonition: your node is enrolled, jobs are running, and results are captured. Next steps: explore [Foundry](../feature-guides/foundry.md) to build custom node images, or [mop-push](../feature-guides/mop-push.md) to push jobs from the CLI.
