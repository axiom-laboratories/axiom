Replace the temporary stub with the full mop-push guide. Structure:

1. **Title and intro**: "# mop-push CLI"
   - One paragraph: mop-push is the command-line tool for pushing signed jobs to Master of Puppets from a developer or CI/CD machine. It handles OAuth authentication, Ed25519 signing, and job submission in a single workflow.
   - `!!! note "Prerequisites"` admonition: "This guide assumes the stack is running and at least one node is enrolled. If you haven't done that yet, start with [Getting Started](../getting-started/prerequisites.md)."

2. **Install section**: "## Install"
   - Install from the repo root:
     ```bash
     pip install -e .
     ```
   - Verify: `mop-push --help`
   - `!!! tip "pipx for isolation"` admonition: for operators who don't want to pollute their global Python env, `pipx install -e .` installs mop-push in an isolated environment.

3. **Login section**: "## Login"
   - Intro sentence: mop-push uses the OAuth 2.0 Device Authorization Grant (RFC 8628) — no browser on the machine running the CLI is required.
   - Set the server URL (either flag or env var):
     ```bash
     export MOP_URL=https://your-host
     ```
   - Run login:
     ```bash
     mop-push login
     ```
   - Show the verified CLI output (exact format from interfaces block above)
   - Step-by-step explanation of what happens: user code is shown → open the browser URL → approve the request in the dashboard → CLI polls and receives the token
   - Credential storage: `!!! info "Credential store"` admonition: credentials are saved to `~/.mop/credentials.json` with permissions `0600`. If running as root or in a container, confirm the home directory is writable and the path is what you expect.
   - Session lifetime note: credentials expire after the JWT lifetime configured on the server (default: 15 minutes for access tokens). Run `mop-push login` again to re-authenticate.

4. **Ed25519 key setup section**: "## Ed25519 Key Setup"
   - Why signing is required: all jobs submitted via mop-push must be signed with an Ed25519 private key. The public key must be registered in the dashboard before pushing.
   - **Generate a keypair** — two options:
     - Using openssl (any machine with openssl installed):
       ```bash
       openssl genpkey -algorithm ed25519 -out signing.key
       openssl pkey -in signing.key -pubout -out verification.key
       ```
     - Using admin_signer.py (if the toms_home tooling is available):
       ```bash
       python ~/Development/toms_home/.agents/tools/admin_signer.py --generate
       # Creates: secrets/signing.key and secrets/verification.key
       ```
   - `!!! danger "Protect your private key"` admonition: never commit `signing.key` to a git repository. Add it to `.gitignore`. In production, store private keys in a secrets manager (Vault, AWS Secrets Manager, or similar) and inject them via environment variable or mounted volume.
   - **Register the public key in the dashboard**:
     - Go to **Signatures** in the dashboard sidebar
     - Click **Add Signature Key**
     - Paste the contents of `verification.key`
     - Give it a descriptive name (e.g., `ci-deploy-key`)
     - Save — note the **Key ID** shown in the list. You will pass this as `--key-id` when pushing jobs.

5. **Push a job section**: "## Push a Job"
   - Write a simple test script:
     ```python
     # hello.py
     print("Hello from mop-push!")
     import platform
     print(f"Node: {platform.node()}, OS: {platform.system()}")
     ```
   - Push it:
     ```bash
     mop-push job push \
       --script hello.py \
       --key signing.key \
       --key-id <your-key-id>
     ```
   - Expected output: the CLI prints the created job ID and its status: `DRAFT`
   - `!!! warning "DRAFT jobs are not dispatched"` admonition: a newly pushed job has status DRAFT. Nodes will not receive DRAFT jobs. You must publish the job via the dashboard or the API before it is dispatched. See the next section.

6. **Publish from Staging section**: "## Publish from Staging"
   - After pushing, go to the dashboard **Jobs → Staging** tab
   - Your job appears with status **DRAFT** and a preview of the script content
   - Review the script in the staging view — this is your gate to confirm the right script and signature are present before dispatching to nodes
   - Click **Publish** to promote the job to **ACTIVE**
   - Active jobs are dispatched to nodes matching the target tags (or any available node if no tags are set)
   - Monitor progress: the job moves through **PENDING → ASSIGNED → COMPLETED** in the Jobs list
   - Click the job to view the captured output

7. **Updating an existing job section**: "## Updating a Job"
   - If you push a new version of a script and want to replace an existing job:
     ```bash
     mop-push job push \
       --id <existing-job-id> \
       --script updated_script.py \
       --key signing.key \
       --key-id <your-key-id>
     ```
   - This creates a new DRAFT version associated with the existing job ID
   - Go to Staging and Publish as before

8. **Environment variable reference** at the bottom — quick reference table:
   | Variable | Purpose | Example |
   |----------|---------|---------|
   | `MOP_URL` | Base URL for the MoP server | `https://your-host` |
