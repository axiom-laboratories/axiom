## Conda Mirror Backend Pattern

From Phase 111 (npm/NuGet/OCI mirrors), the established pattern for new mirrors:
1. Add `_mirror_{ecosystem}()` method to mirror_service.py using throwaway container
2. Use asyncio.to_thread() for subprocess calls (non-blocking)
3. Update Caddyfile with new path for serving packages
4. Add env var (e.g., CONDA_MIRROR_URL) to .env.example
5. Integrate into foundry_service for ecosystem-aware injection

## Existing Mirror Methods

From mirror_service.py:
- `_mirror_pypi()` — dual-platform downloads via pip wheels
- `_mirror_apt()` — APT packages via apt-get download in debian:12-slim
- `_mirror_apk()` — Alpine packages via apk fetch in alpine:3.20
- `_mirror_npm()` — npm packages via npm install in node:20-alpine

- `get_nuget_config_content()` — NuGet.config for NuGet

## Conda Specifics

**Channel structure:** Anaconda uses `{channel}/{platform}/` subdirectories (e.g., `conda-forge/linux-64/`, `defaults/linux-x86_64/`).

**repodata.json:** Index file listing all packages in a channel. Regenerated via `conda index` command inside a throwaway miniconda container.

**Storage:** `mirror-data/conda/{channel}/{subdir}/` matches Conda's native directory layout for serving via static HTTP (Caddy).

**.condarc format:** YAML file with `channels:` list (order matters — first match wins). Injected into Foundry-built images.

**ToS context:** Anaconda's `defaults` channel requires commercial license for organizations with 200+ employees. conda-forge is free community alternative.

Add `_mirror_conda(ingredient: ApprovedIngredient, version_constraint: str) -> bool` method to MirrorService class. Pattern:

1. Extract channel from ingredient.mirror_path (e.g., "conda-forge" or "defaults" or custom URL)
2. Extract version from version_constraint (format: `package-name==X.Y.Z`)
3. Launch throwaway miniconda:latest container with:
   - Volume mount: `/mirror-data/conda` → `mirror-data/conda/` on host
   - Command: `conda create --download-only -c {channel} -p /tmp/conda-env {package}=={version}`
4. After download, run `conda index {channel_dir}` inside same container to regenerate repodata.json
5. Use asyncio.to_thread(subprocess.run, ...) for non-blocking execution (120s timeout)
6. Update ingredient.mirror_status to MIRRORED on success, FAILED on error
7. Catch subprocess errors, log to ingredient.mirror_log, return False on failure

Handle edge cases:
- Custom channel URLs (ingredient.mirror_path contains http://) — pass as `-c {url}` to conda create
- Platform detection — Conda uses platform tags (linux-64, linux-aarch64, osx-64, win-64, etc.) — directory already provided by Foundry
- Missing channel — return 422 Unprocessable Entity from caller (check in add_ingredient)

Pattern reference from _mirror_npm() and _mirror_nuget() — same asyncio.to_thread pattern.

Requirements.txt: Ensure miniconda is available (it's in the base image). If needed, add `conda>=23.0` or equivalent.
