3. **Cascade Cancellation** (15-20 lines):
   - "## Cascade Cancellation (Failure Propagation)"
   - "When a step fails, the system automatically cancels all downstream steps UNLESS the failure is isolated by a gate:"
   - Example 1: "Linear pipeline A→B→C. A fails → B and C are CANCELLED. Workflow status: FAILED."
   - Example 2: "Pipeline with IF gate: A→[IF gate]→B (success branch), Rollback (failure branch)→C. A fails → IF gate takes failure branch (Rollback runs) → C runs normally. Workflow status: PARTIAL."

   - "## Gate Execution Semantics"
   - Subsections for each gate type:
     - **IF_GATE**: "Evaluates conditions against `result.json` from upstream step. If ALL conditions match, takes primary branch. If any condition fails, takes failure branch. Routes to exactly one branch (never both)."
     - **AND_JOIN**: "Waits for ALL incoming branches to complete before releasing downstream steps. Propagates failure from any branch (unless isolated by an IF gate above it)."
     - **OR_GATE**: "Releases downstream steps as soon as ANY incoming branch completes. Other branches are CANCELLED (they're not needed)."
     - **PARALLEL**: "Fans out to multiple independent downstream branches; all execute concurrently. No waiting; just releases all children."
     - **SIGNAL_WAIT**: "Pauses execution until an external signal (named) is posted via the existing Signal mechanism. When signal arrives, continues."
   - Key insight: "Gates don't execute jobs; they control flow topology. PARALLEL doesn't parallelize a single job — it releases multiple downstream steps for concurrent execution."

4. **BFS Wave Dispatch Algorithm** (30-40 lines):
   - "## Dispatch Algorithm (BFS Waves)"
   - High-level explanation (5 lines): "The dispatch algorithm is a breadth-first search through the DAG. Each 'wave' consists of all steps with all dependencies complete. The algorithm ensures steps dispatch in topological order without explicit ordering."
   - Pseudocode / explanation (20-30 lines):
   ```
   function dispatch_workflow_run(workflow_id, workflow_run_id):
       # Initialize: mark all steps with zero incoming edges as ready
       pending_steps = get_all_steps_for_workflow(workflow_id)
       wave = 1
       
       while pending_steps exists:
           # Find all steps with all dependencies completed
           wave_steps = find_steps_with_all_deps_complete(pending_steps)
           
           # Dispatch this wave concurrently
           for each step in wave_steps:
               job = create_job_for_step(step, workflow_run_id)
               mark_step_as_ASSIGNED(step)
           
           # Wait for at least one to complete (polling loop)
           while any step in wave is ASSIGNED or RUNNING:
               poll_job_status()
               when job completes:
                   mark_step as SUCCEEDED or FAILED
                   if FAILED and not isolated by gate:
                       cascade_cancel(downstream_steps)
                   update_workflow_run_status()
           
           wave += 1
       
       finalize_workflow_run_status()
   ```
   - Explanation (5-8 lines): "The algorithm respects the DAG topology: a step is only dispatched when ALL its incoming edges have completed. This guarantees correct ordering without explicit sequencing. Multiple steps in the same wave are independent and could execute in any order."

5. **Concurrency Safety: SELECT...FOR UPDATE (CAS Guards)** (15-20 lines):
   - "## Concurrency Safety"
   - "Problem: Multiple poll cycles might simultaneously detect that a step is ready to dispatch, causing duplicate Job creation"
   - "Solution: Atomic row-level locks using SELECT...FOR UPDATE"
   - Code pattern (5-8 lines):
   ```python
   # In workflow_service.py::dispatch_workflow_run()
   stmt = select(WorkflowRun).where(...).with_for_update()
   row = session.execute(stmt).scalar_one()
   # At this point, row is locked; other transactions wait
   # Check again: is this step actually ready to dispatch?
   if is_ready_to_dispatch(row):
       create_job()
       mark_ASSIGNED()
   # On commit, lock is released
   ```
   - Impact (5-8 lines): "This ensures that if two poll cycles run simultaneously, only one will succeed in creating the Job. The second will re-check and see the step is already ASSIGNED, skipping duplicate creation. This is critical for correctness under concurrent load."

6. **Cascade Cancellation Logic** (15-20 lines):
   - "## Cascade Cancellation (Failure Propagation)"
   - "When a step fails, the algorithm cascades the failure to all downstream PENDING steps UNLESS the failure is isolated by a gate"
   - Code pattern (5-8 lines):
   ```python
   # In workflow_service.py::cascade_cancel()
   if step.failed and not is_isolated_by_gate(step):
       for downstream_step in find_downstream_steps(step):
           if downstream_step.status == PENDING:
               mark_CANCELLED(downstream_step)
               cascade_cancel(downstream_step)  # Recursive
   ```
   - Isolation gates (5-8 lines): "An IF_GATE with a failure branch is an isolation point. If the upstream step fails, the IF gate routes to the failure branch; the primary branch is cancelled, but the failure branch executes. The failure is absorbed; downstream steps don't cascade."
   - Clarification (3-5 lines): "AND_JOIN, OR_GATE, and PARALLEL also break cascade chains (they merge branches), but IF_GATE is the primary isolation mechanism."

7. **Lazy Import Pattern (Circular Dependencies)** (10-15 lines):

   - Impact (2-3 lines): "This defers the import until the function runs, avoiding the circular dependency at parse time. Contributors should be aware of this pattern when adding new cross-module calls."
