
3. **Common Operator Tasks** (20-25 lines):
   - "## Common Tasks"
   - "### Checking if a workflow is stuck"
     - Steps: Fetch `/api/workflows/{id}/runs/{run_id}` → check status (RUNNING, COMPLETED, PARTIAL, FAILED, CANCELLED)
     - If RUNNING: check timestamp (started_at); if > 1 hour old, may indicate a hung step
     - Solution: Cancel via `DELETE /api/workflows/{id}/runs/{run_id}` and investigate logs
   - "### Viewing a failed step's output"
     - Dashboard: WorkflowRunDetail → click red step → Step Drawer shows logs and result.json
     - Or API: `GET /api/executions/{job_guid}/logs` (job_guid is in WorkflowStepRun record)
   - "### Understanding PARTIAL status"
     - PARTIAL = expected; an IF gate isolated a failure
     - Check the DAG: look for gates that took the failure branch
     - Example: "Step A fails → IF gate routes to failure handler Step B → Step C runs normally → Result: PARTIAL (not FAILED)"
   - "### Monitoring cron schedules (Phase 149)"
     - Cron is managed by APScheduler
     - Next run is computed from schedule_cron expression; visible in Workflows list
     - To verify a cron ran: check /api/workflows/{id}/runs for recent entries with trigger_type=CRON
   - "### Testing a webhook"
     - Create webhook: `POST /api/workflows/{id}/webhooks` → save secret
     - Test: Use Python example from API docs to send signed POST to `/api/webhooks/{webhook_id}/trigger`
     - Verify: Check /api/workflows/{id}/runs for a new run with trigger_type=WEBHOOK

4. **Troubleshooting** (20-25 lines):
   - "## Troubleshooting"
   - "### Symptom: Workflow never progresses (stuck in RUNNING)"
     - **Likely cause**: Step is hung waiting for a job to complete
     - **Debug**: Check /api/workflows/{id}/runs/{run_id}; find ASSIGNED/RUNNING step; inspect its job logs
     - **Fix**: Cancel the run; investigate the step's code/job definition
   - "### Symptom: Workflow marked FAILED when expected PARTIAL"
     - **Likely cause**: Cascade cancellation didn't respect an IF gate isolation
     - **Debug**: View the DAG; trace which step failed and which gate should have isolated it
     - **Fix**: Check the gate's failure branch is defined (config_json not empty); check step edges are correct
   - "### Symptom: Webhook events rejected (400 Bad Request)"
     - **Likely cause**: Signature verification failed (bad HMAC, stale timestamp, replayed nonce)
     - **Debug**: Check X-Webhook-Signature header matches HMAC-SHA256(secret, payload + timestamp)
     - **Fix**: Verify timestamp is recent (within ±5 min); check nonce hasn't been used (24h dedup); regenerate secret if needed
   - "### Symptom: Parameters not injected into step jobs"
     - **Likely cause**: WorkflowParameter rows not created or parameters_json not in WorkflowRun
     - **Debug**: Check /api/workflows/{id} for parameters array; check POST /api/workflows/{id}/runs includes parameters dict
     - **Fix**: Ensure workflow was created with parameters array; ensure manual trigger includes parameter values
   - "### Symptom: IF gate took wrong branch"
     - **Likely cause**: Condition evaluation logic error or wrong path in result.json
     - **Debug**: Check the IF gate's config_json (conditions); check the upstream step's result.json output
     - **Fix**: Verify operator and path match the step's actual output structure; update condition if needed

5. **Recovery Procedures** (10-15 lines):
   - "## Recovery"
   - "### Restarting a failed workflow"
     - "Current: Manual re-trigger via `POST /api/workflows/{id}/runs` (Phase 151 UI coming)"
     - "Planned (v24.0): Rerun from failure point (restart from first failed step)"
   - "### Clearing stuck steps"
     - "Cancel the entire run via `DELETE /api/workflows/{id}/runs/{run_id}`"
     - "All ASSIGNED/RUNNING steps will be aborted; all PENDING steps will be marked CANCELLED"
     - "Workflow marked CANCELLED"
   - "### Resetting a webhook secret"
     - "Webhook secrets are one-time reveal; if lost, delete and recreate: `DELETE /api/workflows/{id}/webhooks/{webhook_id}` then `POST /api/workflows/{id}/webhooks`"

6. **Monitoring Best Practices** (8-12 lines):
   - "## Monitoring"
   - "**Real-time**: Use WebSocket `/ws?token=<jwt>` to subscribe to workflow_run_updated and workflow_step_updated events"
   - "**Polling**: `GET /api/workflows/{id}/runs` every 10-30s for status updates"
   - "**Dashboard**: Workflows list shows all runs; WorkflowRunDetail shows live DAG with status overlay"
   - "**Alerting**: Monitor status changes via WebSocket or polling; alert on FAILED runs"

Total target: ~90-110 lines
