# Plan 85-02: Docs and Homepage Integration

## Objective

Create the screenshot asset directories, update getting-started and feature guide docs to embed screenshots, add a "See it in action" section to the marketing homepage, and remove the outdated `generate_screenshots.py`.

## Context

- Screenshots will be committed separately by an operator after running `capture_screenshots.py` — these tasks establish the *structure* (directories + image references) so the docs render correctly once PNGs exist
- Use standard markdown inline images — no MkDocs-specific syntax
- Relative path from `getting-started/` and `feature-guides/` to assets: `../assets/screenshots/`
- Homepage section uses plain HTML + CSS consistent with existing `homepage/style.css` design language

## must_haves

- `docs/docs/assets/screenshots/` directory exists in git (via `.gitkeep`)
- `homepage/assets/screenshots/` directory exists in git (via `.gitkeep`)
- `docs/docs/getting-started/enroll-node.md` contains image references for `nodes.png` and `node_detail.png`
- `docs/docs/getting-started/first-job.md` contains image references for `jobs.png` and `job_detail.png`
- `homepage/index.html` contains a "See it in action" section with `<img>` tags for 4 key screenshots
- `puppeteer/dashboard/generate_screenshots.py` is deleted (replaced by `tools/capture_screenshots.py`)

## Notes

- Screenshots referenced in docs won't render until an operator runs `tools/capture_screenshots.py` and commits the PNG files. This is intentional — the docs structure is committed first, screenshots populated separately.
- `loading="lazy"` on `<img>` tags is good practice for a docs/marketing page with potentially large screenshots.
- The homepage section uses 4 screenshots: `dashboard.png`, `nodes.png`, `jobs.png`, `audit.png`. The audit screenshot was chosen over `foundry.png` to emphasize the security/compliance angle for the marketing audience.
