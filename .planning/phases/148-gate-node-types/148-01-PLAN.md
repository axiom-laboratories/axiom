Create a new service file with gate-specific logic. Implement the following methods (following patterns from RESEARCH.md code examples):

1. `resolve_field(data: Dict, path: str) -> Tuple[bool, Any]` — Resolve dot-path in data dict (e.g., "exit_code", "data.status"). Return (found, value). Handle missing paths gracefully.

2. `evaluate_condition(condition: Dict, result: Dict) -> bool` — Evaluate a single condition: {"field": "...", "op": "eq|neq|gt|lt|contains|exists", "value": ...}. Return True if condition matches, False otherwise. Handle type mismatches gracefully (catch TypeError, return False).

3. `evaluate_conditions(conditions: List[Dict], result: Dict) -> bool` — Evaluate all conditions in list (AND logic). Return True only if all conditions match.

4. `evaluate_if_gate(config_json: str, result: Dict) -> Tuple[Optional[str], Optional[str]]` — Evaluate IF gate config. config_json = '{"branches": {"true": [...conditions...], "false": [...conditions...]}}'. Evaluate "true" branch first (convention), then "false". Return (branch_taken, error_message). branch_taken in ("true", "false"); error_message if no match.

Operators to support:
- `eq` — actual == value
- `neq` — actual != value
- `gt` — actual > value
- `lt` — actual < value
- `contains` — str(value) in str(actual)
- `exists` — field exists in data (always True if resolve_field returns found=True)

Implementation notes:
- Use `json.loads()` to parse config_json; handle JSONDecodeError gracefully
- For numeric comparisons (gt, lt), catch TypeError if value is non-numeric and return False
- For 'exists' operator, ignore the value field; just check if field was found
- All methods should be static (no instance state needed)
