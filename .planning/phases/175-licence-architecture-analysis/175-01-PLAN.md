**SECTION 5: Future Architecture Wireframe — VPS Licence Server**
Based on the VPS investigation todo (D-02, D-03), provide a detailed wireframe for the post-launch VPS server:

Sub-sections:

1. **Design Principles**:
   - Soft enforcement: check-in failures do not break customers (7-day buffer per D-03)
   - Privacy-first: deployment_id is hashed (hash(licence_id + hostname_hash)), not raw hostname
   - Backwards compatible: air-gapped customers unaware server exists
   - Minimal: focus on online-tier check-in and revocation, not full licence portal

2. **Two-Tier Licence Model**:
   ```
   Online Tier (deployment_mode: online):
   - TTL: 30–90 days (recommend 60d default)
   - Auto-renewal: check-in every 7 days (recommend)
   - Revocation: YES (remotely revocable, instant effect)
   - Check-in required: YES (but non-blocking; 7-day buffer on failure)
   - Use case: standard customers, connected deployments
   
   Air-Gapped Tier (deployment_mode: airgapped):
   - TTL: 1–3 years (recommend 2y default, expires on fixed date)
   - Auto-renewal: NO (long TTL, customer renews manually near expiry)
   - Revocation: NO (no check-in, no revocation path)
   - Check-in required: NO (server never contacted)
   - Premium pricing: reflects reduced vendor visibility + no revocation
   - Use case: isolated/classified networks, offline deployments
   ```

3. **VPS Infrastructure**:
   - **Hosting**: Hetzner CX11 (€5–10/mo) or Fly.io free tier with backup paid option
   - **Stack**: FastAPI (Python 3.11), Caddy (TLS), SQLite (dev) or Postgres (prod)
   - **Schema**: minimal — 4 tables (Licence, Deployment, CheckIn, RevokedCert)
   - **TLS**: Let's Encrypt via Caddy (same pattern as main MoP server)
   - **Backup**: daily snapshots to S3 or GitHub (licence data is non-critical but auditable)

4. **API Surface**:
   ```
   POST /register
     Request: {licence_id, customer_id, issued_at}
     Response: {deployment_id, registered_at, next_checkin_due}
     Purpose: first-time activation, create deployment registry entry
   
   POST /checkin
     Request: {licence_id, deployment_id, server_version, node_count, checked_at}
     Response: {valid, revoked, message, grace_days_remaining (if GRACE), retry_after_seconds}
     Purpose: periodic heartbeat (online tier only)
   
   GET /status/{licence_id}
     Auth: admin API key
     Response: {status: ACTIVE|REVOKED|EXPIRED, deployed_count, first_seen, last_seen}
     Purpose: admin visibility
   
   POST /revoke/{licence_id}
     Auth: admin API key
     Request: {reason} (optional)
     Response: {revoked: true, revoked_at}
     Purpose: admin revocation (marks licence as revoked, instant effect on next check-in)
   ```

5. **Client-Side Integration** (changes to `licence_service.py`):
   ```python
   # Optional env var (absence = air-gapped mode)
   AXIOM_LICENCE_SERVER_URL = os.getenv("AXIOM_LICENCE_SERVER_URL", None)
   
   # In load_licence() after JWT is validated:
   if payload.get("deployment_mode") == "online" and AXIOM_LICENCE_SERVER_URL:
       # Fire-and-forget check-in (async, 5s timeout, no blocking)
       asyncio.create_task(checkin_licence_server(licence_id, server_url, timeout=5))
   
   # On check-in response:
   if response.get("revoked"):
       # Degrade to CE mode immediately
       broadcast_websocket_event("licence_revoked", {licence_id, reason})
       # Running jobs unaffected, new jobs rejected
   ```

6. **Check-In Flow (Non-Blocking)**:
   - Deployment starts with online-tier JWT
   - On startup: emit async check-in POST to VPS (licence_id + hashed deployment_id + version + node_count)
   - VPS responds: {valid, revoked, next_checkin_in_days}
   - If VPS unreachable: log warning, continue normally (7-day buffer before CE degradation)
   - If VPS returns revoked=true: immediately degrade to CE mode
   - Check-in repeats: every 7 days (configurable)

7. **Security Considerations**:
   - **Secrets**: VPS revocation API key stored in `secrets/licence-server-key` on puppeteer
   - **Clock skew**: accept ±5min timestamp tolerance on check-in requests
   - **Rate limiting**: 100 check-ins/min per licence_id (prevent abuse)
   - **Data minimization**: VPS never receives raw hostname, customer subnet, or user data
   - **Breach scenario**: compromised VPS can revoke licences but cannot issue new ones (private key stays in axiom-licenses repo)

8. **Operational Notes**:
   - VPS is optional (air-gapped customers don't need it)
   - Horizontal scaling: VPS can be scaled to multiple instances behind a load balancer (uses Postgres) if needed
   - DR: daily backup to S3; restore from snapshot takes ~5min
   - Monitoring: `/health` endpoint for uptime checks; alert on check-in error rates >5%
   - Sunset path: if VPS taken offline, online-tier customers have 7-day grace before degradation

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Vendor → Customer | Vendor issues a JWT licence; customer deploys it as env var. No further network communication (offline-first). |
| Online-tier check-in → VPS | Optional check-in from deployment to VPS server (non-blocking); VPS is owned by vendor. |
| Admin → VPS revocation API | Admin user revokes a licence on VPS; requires API key authentication. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-175-01 | Tampering | Git repo audit records (YAML files) | mitigate | Git repo is version-controlled and GitHub-hosted; revision history prevents silent tampering. Audit records are read-only after commit. |
| T-175-02 | Information Disclosure | JWT payload (customer_id, features, node_limit) | accept | JWT is base64-encoded, not encrypted. Customer must protect their licence key (env var). No PII in JWT. |
| T-175-03 | Spoofing | Fake VPS server responding to client check-ins (future-state) | mitigate | VPS server uses Let's Encrypt TLS (cert pinning not required for MVP); client validates signature of response before acting. Recommended: client library validates HTTPS cert chain. |
| T-175-04 | Repudiation | Customer denies using a licence (future-state online-tier) | mitigate | VPS deployment_id field records check-in; hashed deployment_id + server_version + node_count provides non-repudiation of deployment. Git audit record + check-in log = full chain. |
| T-175-05 | Denial of Service | VPS licence server down; online-tier deployments blocked (future-state) | mitigate | Client check-in is non-blocking with 7-day grace buffer (D-03); deployments continue if VPS unreachable. Soft enforcement prevents hard outage. |
| T-175-06 | Elevation of Privilege | Attacker gains access to VPS revocation API key | mitigate | API key stored in secrets/licence-server-key on puppeteer (not in Git). Rate limiting on revocation API (10 req/min). MFA for admin access to puppeteer. |

**Security Enforcement:** Enabled (all EE phases include STRIDE analysis per CLAUDE.md).

Note: This analysis phase does not introduce new code or attack surface. The threat model documents future risks introduced by the VPS licence server architecture (Phase LIC-IMPL-01).
