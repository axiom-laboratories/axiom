# Phase 15 Context: Smelt-Check, BOM & Lifecycle

## Goal
Implement automated post-build validation ("Smelt-Check"), deep Bill of Materials (BOM) tracking, and strict image lifecycle governance. This ensures that only verified, secure images are deployed to the fleet and that old or compromised images are gracefully phased out.

## Decisions

### 1. Smelt-Check (Post-Build Validation)
- **Staging Environment**: Smelt-Checks will run on a dedicated "Staging Node" with hard resource limits (memory/CPU).
- **Interactive Failure**: If a check fails, the system will pause and ask the admin whether to keep the image for debugging or delete it.
- **Comprehensive Scope**: Checks include `validation_cmd` execution, Puppet agent startup verification, and mTLS connectivity tests.
- **Friction-based Bypass**: Admins can skip Smelt-Check for emergency fixes, but must type "OVERRIDE" in a confirmation dialog.

### 2. BOM Structure & Provenance
- **Runtime Capture**: The canonical BOM will be generated by running `pip list --json` and `dpkg-query` inside the staging node during Smelt-Check.
- **Full Provenance**: Includes `sha256` hashes of all mirrored ingredients.
- **Global Search**: A dedicated "BOM Explorer" view will allow admins to search for images by package name and version across the entire fleet.
- **Standardized Export**: Support for exporting BOMs in SPDX-compatible JSON format.

### 3. Lifecycle Enforcement & Auto-Pruning
- **Graceful Revocation**: REVOKED images will allow current jobs to finish but block all new job assignments.
- **Enrollment Lock**: Nodes running REVOKED images are blocked from enrolling or heartbeating.
- **Proactive Deprecation**: Nodes on DEPRECATED images display a "Rebuild Recommended" warning in the dashboard.
- **Intelligent Pruning**: Auto-delete DEPRECATED images after an admin-defined TTL (e.g., 30 days), but ONLY after the last node using that image has stopped heartbeating.

### 4. Validation & Visibility UX
- **Real-time Build Progress**: The UI will show a progress bar specifically for the Smelt-Check phase.
- **Detailed Failure Reports**: Full stdout/stderr from the staging node validation will be viewable in the dashboard.
- **Permission Gates**: Override and Revocation actions require `foundry:admin` (or equivalent high-level) permissions.

## Code Context
- **Staging Runner**: Implement a `StagingService` that manages the lifecycle of the ephemeral validation container.
- **BOM Index**: Add a `package_index` table to the database for global searching across BOMs.
- **Lifecycle Worker**: A periodic task to handle auto-pruning and status transitions.

## Success Criteria
1. Newly built images automatically transition to `ACTIVE` only after passing Smelt-Check.
2. A searchable BOM is available for every image built via the Foundry.
3. Nodes running `REVOKED` images are successfully blocked from the Control Plane.
4. Auto-pruning correctly identifies and deletes unused, expired images.
