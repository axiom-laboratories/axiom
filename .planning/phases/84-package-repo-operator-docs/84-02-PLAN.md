    Create docs/docs/runbooks/package-mirrors.md. Structure: front-matter-free markdown with H1 title and three H2 sections. No verbose prose — setup + verify format throughout. Follow node-validation.md formatting conventions (fenced code blocks, numbered steps, admonition-style notes).

    **H1:** `# Package Mirror Setup`

    Short intro paragraph (2 sentences): explains that package mirrors allow Foundry node image builds to resolve packages from internal sidecars instead of the public internet, and that this runbook covers devpi (PyPI), apt-cacher-ng (APT), and BaGet (PWSH/NuGet).

    ---

    **H2: `## devpi — PyPI Mirror (PKG-01)`**

    1. One paragraph: devpi is a full PyPI caching proxy. It mirrors pypi.org and caches packages locally. devpi is already included in `compose.server.yaml` at port 3141 — no additional compose changes are required.

    2. **Enable the mirror** (numbered steps):
       1. Configure the agent to point Foundry builds at devpi by setting `PYPI_MIRROR_URL`:
          ```bash
          # Option A: environment variable in compose.server.yaml agent service
          environment:
            - PYPI_MIRROR_URL=http://devpi:3141/root/pypi/+simple/
          ```
          Or via API:
          ```bash
          curl -X PATCH https://localhost:8001/admin/mirror-config \
            -H "Authorization: Bearer $TOKEN" \
            -H "Content-Type: application/json" \
            -d '{"pypi_mirror_url": "http://devpi:3141/root/pypi/+simple/"}'
          ```
       2. Rebuild all Foundry templates so the updated `pip.conf` is baked into new node images.

    3. **Seed packages for offline use** (numbered steps using `docker exec`):
       ```bash
       # Create the root/pypi caching index (first-run only)
       docker exec puppeteer-devpi-1 devpi use http://localhost:3141
       docker exec puppeteer-devpi-1 devpi login root --password=""
       docker exec puppeteer-devpi-1 devpi index -c root/pypi bases=root/pypi type=mirror mirror_url=https://pypi.org/simple/
       # Trigger initial cache fill (packages download on first install request)
       docker exec puppeteer-devpi-1 devpi install requests
       ```

    4. **Verify with the PKG-04 validation job**:
       ```bash
       axiom-push job push \
         --script tools/example-jobs/validation/verify_pypi_mirror.py \
         --key signing.key --key-id <your-key-id> \
         --env PYPI_MIRROR_HOST=devpi:3141
       ```
       Expected output on PASS:
       ```
       === Axiom PyPI Mirror Validation ===
       Testing mirror: devpi:3141
       PASS: pip resolved 'requests' from internal mirror (devpi:3141).
             Downloading http://devpi:3141/root/pypi/+f/abc/requests-2.31.0-py3-none-any.whl (62 kB)
       ```
       Expected output on FAIL (mirror not seeded or wrong URL):
       ```
       FAIL: pip is resolving from pypi.org, not the internal mirror.
       ```

    5. **Common issues** (bullet list):
       - **"No matching distribution found"** — using wrong index path. The correct URL is `http://devpi:3141/root/pypi/+simple/`, not `http://devpi:3141/+simple/` (root index is empty).
       - **pip SSL error on HTTP URL** — `trusted-host` must match the devpi hostname. The `MirrorService.get_pip_conf_content()` sets this automatically from the URL.
       - **devpi `root/pypi` index not found after fresh deploy** — run the `devpi index -c root/pypi` init commands above on first start.

    ---

    **H2: `## apt-cacher-ng — APT Mirror (PKG-02)`**

    1. One paragraph: apt-cacher-ng is a transparent caching proxy for APT packages. Adding it as a sidecar lets Foundry builds install Debian/Ubuntu packages through the cache; on subsequent builds and in air-gapped environments, packages resolve from the cache rather than the internet.

    2. **Add the sidecar to compose.server.yaml**:
       ```yaml
       apt-mirror:
         image: sameersbn/apt-cacher-ng:latest
         restart: unless-stopped
         ports:
           - "3142:3142"
         volumes:
           - apt-cache-data:/var/cache/apt-cacher-ng
         environment:
           - DISABLE_IPFORWARD_CHECK=1
       ```
       Also add `apt-cache-data:` under the top-level `volumes:` key.

    3. **Configure Foundry builds to use the proxy** — the apt proxy must be injected at build time and removed from the final image. In any Foundry DEBIAN Blueprint, add a `setup_commands` entry or note in the runbook that the foundry_service.py Dockerfile pattern already injects a proxy line automatically when `APT_PROXY_URL` is set:
       ```bash
       # Set via API
       curl -X PATCH https://localhost:8001/admin/mirror-config \
         -H "Authorization: Bearer $TOKEN" \
         -H "Content-Type: application/json" \
         -d '{"apt_mirror_url": "http://apt-mirror:3142"}'
       ```
       The resulting Dockerfile pattern (auto-generated by Foundry):
       ```dockerfile
       RUN echo 'Acquire::http::Proxy "http://apt-mirror:3142";' > /etc/apt/apt.conf.d/01proxy \
           && apt-get update \
           && apt-get install -y <packages> \
           && rm /etc/apt/apt.conf.d/01proxy
       ```

    4. **Verify** — trigger a Foundry build and check apt-cacher-ng logs:
       ```bash
       docker logs puppeteer-apt-mirror-1 --tail 20
       ```
       Cache hits appear as `200 (ca)` in the log. Cache misses (first build) appear as `200`.

    5. **Common issues** (bullet list):
       - **apt-mirror hostname not resolving from inside Foundry build container** — ensure `apt-mirror` service is in the same Docker network as the agent (`puppeteer_default`).
       - **Proxy baked into final node image** — always `rm /etc/apt/apt.conf.d/01proxy` in the same `RUN` layer as the install. Node APT commands will fail outside the compose network if the proxy line persists.
       - **DISABLE_IPFORWARD_CHECK=1 required** — without this, apt-cacher-ng may fail to start in Docker due to missing IP forwarding.

    ---

    **H2: `## BaGet — PowerShell Module Mirror (PKG-03)`**

    1. One paragraph: BaGet is a lightweight NuGet v3 server compatible with PowerShell's `Install-Module` and `Install-PSResource` commands. Adding it as a sidecar lets PWSH job scripts install modules from an internal repository rather than PSGallery.

    2. **Add the sidecar to compose.server.yaml**:
       ```yaml
       baget:
         image: loicsharma/baget:latest
         restart: unless-stopped
         ports:
           - "5555:80"
         volumes:
           - baget-data:/var/baget
         environment:
           - ApiKey=AXIOM-INTERNAL-NUGET-KEY
           - Storage__Type=FileSystem
           - Storage__Path=/var/baget
           - Database__Type=Sqlite
           - Database__ConnectionString=Data Source=/var/baget/db/baget.db
           - Search__Type=Database
       ```
       Also add `baget-data:` under the top-level `volumes:` key. BaGet is available at `http://baget:5555`.

    3. **Seed a module (example: Pester)**:
       ```bash
       # Download Pester from PSGallery
       curl -L "https://www.powershellgallery.com/api/v2/package/Pester" -o Pester.nupkg

       # Upload to BaGet
       curl -X PUT "http://localhost:5555/api/v2/package" \
         -H "X-NuGet-ApiKey: AXIOM-INTERNAL-NUGET-KEY" \
         -F "package=@Pester.nupkg"
       ```
       Verify via BaGet web UI at `http://localhost:5555` — Pester should appear in the package list.

    4. **Configure a Foundry Blueprint** so node images have the PSRepository pre-registered. Add to the Blueprint's PowerShell setup script:
       ```powershell
       Register-PSRepository `
         -Name "AxiomInternal" `
         -SourceLocation "http://baget:5555/v3/index.json" `
         -PublishLocation "http://baget:5555/v3/index.json" `
         -InstallationPolicy Trusted
       ```

    5. **Verify with a PWSH job script**. Dispatch a job containing:
       ```powershell
       # Prefer Install-PSResource (PowerShellGet v3, available in PS 7.4+)
       if (Get-Command Install-PSResource -ErrorAction SilentlyContinue) {
           Install-PSResource -Name Pester -Repository AxiomInternal -TrustRepository
       } else {
           Install-Module -Name Pester -Repository AxiomInternal -Force
       }
       Import-Module Pester
       Write-Host "=== PASS: Pester $((Get-Module Pester).Version) installed from AxiomInternal ==="
       ```
       Expected output:
       ```
       === PASS: Pester 5.x.x installed from AxiomInternal ===
       ```

    6. **Common issues** (bullet list):
       - **"Unable to find module Pester"** on `Install-Module` — PowerShellGet v2 has partial NuGet v3 compatibility issues. Use `Install-PSResource` (PSResourceGet/PowerShellGet v3) on PowerShell 7.4+ nodes. Check BaGet package list at `http://localhost:5555` to confirm Pester is seeded.
       - **Register-PSRepository already exists** — if the repository was registered in a previous build or run, use `Set-PSRepository -Name AxiomInternal -InstallationPolicy Trusted` or add an `-ErrorAction SilentlyContinue` guard.
       - **BaGet container unhealthy on first start** — the SQLite DB directory `/var/baget/db/` is created on first run. If the volume is empty, wait 10 seconds after `docker compose up` before seeding.

    ---

    **Enterprise alternatives note** (after BaGet section, before end of file):
    Short paragraph: For larger organisations, Sonatype Nexus Repository and JFrog Artifactory both expose NuGet v3 feeds compatible with `Register-PSRepository`. The same `SourceLocation` and `InstallationPolicy Trusted` registration pattern applies — replace `http://baget:5555/v3/index.json` with the Nexus or Artifactory NuGet feed URL.
