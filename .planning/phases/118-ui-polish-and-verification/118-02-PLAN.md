  <name>Task 1: Audit and standardize density, spacing, and button states across all 9 main dashboard views</name>
  <files>
    puppeteer/dashboard/src/views/Dashboard.tsx

    puppeteer/dashboard/src/views/Admin.tsx
  </files>
  <action>
    Audit each of the 9 main views for consistent styling, spacing, and button usage. Apply these standardizations:

    1. **Density and spacing** — Target: Comfortable balanced density matching SaaS dashboards (Notion, Linear, GitHub). Apply these patterns:
       - Main container padding: `p-4` or `p-6` (not p-2, not p-8)
       - Section spacing: `space-y-4` between major sections
       - Card padding: `p-4` inside cards
       - Table cell padding: `px-4 py-2` (not cramped)
       - List item spacing: `space-y-2` for compact lists, `space-y-3` for relaxed lists
       - Grid gaps: `gap-4` standard, `gap-6` for larger cards

    2. **Button states** — Enforce this pattern across all views:
       - Primary actions (submit, save, create, approve): Use `<Button>` component with default styling. Get automatic hover (opacity-90) + focus ring from component variant. No additional classes needed.
       - Secondary/text-only actions (edit, delete, dismiss): Use subtle styling — `text-muted-foreground hover:text-foreground` or `<Button variant="ghost">`. No background change.
       - Table row actions (pencil, trash icons): Use `<Button variant="ghost" size="sm">` for 24px icon buttons with minimal padding.
       - Confirmation dialogs: Primary action gets full Button styling, Cancel gets variant="outline".

    3. **Replace "Loading..." text with Skeleton loaders** — Identify all loading states in each view:
       - Data-fetching (useQuery loading): Replace `{isLoading && <p>Loading...</p>}` with skeleton matching expected content layout
       - Example for card list: `{isLoading ? <div className="space-y-2"><Skeleton className="h-24 w-full" /><Skeleton className="h-24 w-full" /></div> : <CardList />}`
       - Example for table: `{isLoading ? Array(5).fill(0).map((_, i) => <Skeleton key={i} className="h-8 w-full" />) : <Table />}`

    4. **Standardize dialog sizes** — Review all modals/dialogs:
       - Small (confirmations, alerts): `<DialogContent className="max-w-md">` (400px)
       - Medium (forms, editors): `<DialogContent className="max-w-lg">` (550px)
       - Large (complex editors, multi-step): `<DialogContent className="max-w-2xl">` (700px)
       - Ensure consistent padding inside: `<div className="space-y-4 p-4">`

    5. **Table consistency** — All data tables should:
       - Use shadcn `<Table>` component (already in ui/)
       - Row hover: `hover:bg-muted` for subtle highlight, or `hover:bg-muted/60` if darker is needed
       - Header bold, left-aligned text
       - Numeric columns right-aligned
       - Status badges consistent color scheme from Wave 1

    6. **Apply changes systematically** — For each view:
       - Identify the main layout structure (grid, flex, sections)
       - Normalize padding/margin to p-4/space-y-4 baseline
       - Replace raw buttons with Button component calls (with correct variant)
       - Add skeleton loaders for any {isLoading && <p>} patterns
       - Verify responsive: does the view look reasonable at 1200px and at 768px?

    Example transformation:
    ```tsx
    // Before (inconsistent)
    <div className="p-8">
      {loading ? <p className="text-gray-500">Loading...</p> : <div>Data</div>}
      <button className="bg-blue-500 text-white px-3 py-1">Click</button>
    </div>

    // After (consistent)
    <div className="p-4">
      {loading ? <Skeleton className="h-24 w-full" /> : <div>Data</div>}
      <Button>Click</Button>
    </div>
    ```
  </action>
  <verify>
    - `npm run build` succeeds with no TypeScript errors
    - Each view file compiles without warnings
    - Manual inspection of each view in browser:
      - Dashboard: Cards have consistent spacing, buttons look good, no "Loading..." text visible
      - Nodes: Table rows have subtle hover, stat cards have balanced density
      - Jobs: Job queue renders with skeleton loaders during data fetch
      - JobDefinitions: Dialog sizes are consistent, buttons have hover states
      - Templates: Grid layout consistent, skeleton loaders appear during load
      - Signatures: Table styling matches other tables
      - Users: User list has consistent spacing, edit dialogs are medium-sized
      - AuditLog: Audit log table rows have hover states, timestamp formatting consistent
      - Admin: All admin panels have consistent card styling and button placement
    - Grep for remaining "Loading..." text (should find only legitimate non-UI text): `grep -rn "Loading\.\.\." puppeteer/dashboard/src/views/ --include="*.tsx" | grep -v "TODO\|comment\|// " | wc -l` (should be 0 or minimal)
  </verify>
  <done>

    - All loading states display Skeleton loaders instead of "Loading..." text
    - Dialogs use standard sizes (Small 400px, Medium 550px, Large 700px)
    - Table rows have consistent hover behavior
    - Build succeeds, no TypeScript errors
  </done>

  <files>

  <action>
    1. **Verify Tailwind breakpoints in tailwind.config.js:**
       - Ensure `md` breakpoint is 768px: `screens: { ..., md: '768px', lg: '1024px', ... }`
       - If not present, add to screens object in tailwind.config.js
       - Standard breakpoints: `sm: 640px`, `md: 768px`, `lg: 1024px`, `xl: 1280px`, `2xl: 1536px`

    2. **Sidebar collapse at 768px in MainLayout.tsx:**
       - Current implementation may have sidebar always visible. At 768px (iPad), sidebar should collapse to hamburger menu.
       - Pattern:
         ```tsx
         <div className="flex h-screen">
           {/* Sidebar: hidden on mobile/tablet, visible on desktop */}
           <aside className="w-64 bg-card border-r hidden md:block">
             {/* Sidebar content */}
           </aside>

           {/* Sheet component for mobile sidebar (hamburger menu) */}
           <Sheet>
             <SheetTrigger asChild>
               <Button variant="ghost" size="sm" className="md:hidden">
                 <Menu className="h-6 w-6" />
               </Button>
             </SheetTrigger>
             <SheetContent side="left" className="w-64 p-0">
               {/* Same sidebar content */}
             </SheetContent>
           </Sheet>

           {/* Main content */}
           <main className="flex-1 overflow-auto">
             {/* Page content */}
           </main>
         </div>
         ```
       - Import `Menu` icon from Lucide, `Sheet` + `SheetTrigger` + `SheetContent` from shadcn/ui

    3. **Header responsive behavior:**
       - Header should remain full-width on all breakpoints
       - Breadcrumb or title may need to truncate on small screens
       - Theme toggle stays in header or sidebar footer depending on current layout

    4. **Main content area responsive:**
       - Content should be scrollable vertically (not pushed off screen)
       - Grid layouts should collapse from multi-column to single column at 768px:
         Example: `grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3` (1 col on mobile, 2 on tablet, 3 on desktop)
       - Tables should become horizontally scrollable on small screens or collapse to card view
       - For tables: Add `overflow-x-auto` wrapper at breakpoint if needed

    5. **Modal/Dialog behavior:**
       - On small screens, dialogs should be full-width (minus padding) to use available space
       - Pattern: `<DialogContent className="max-w-lg md:max-w-2xl">` (full-width on mobile, constrained on desktop)

    6. **Test responsive:** Use browser DevTools device emulation to test at:
       - 375px (iPhone SE)
       - 768px (iPad)
       - 1024px (iPad Pro)
       - 1400px (desktop)
       Verify: sidebar toggles off/on correctly, content remains readable, no horizontal overflow.
