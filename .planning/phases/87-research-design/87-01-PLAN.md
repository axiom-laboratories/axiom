**Section 1 content (RSH-01):**
- State that the competitor pain points report was reviewed (`mop_validation/reports/plans/20262903/competitor_pain_points.md`)
- Map each competitor pain point to its v16.0 feature:
  - Airflow/Temporal/Prefect report "success" while execution state is wrong → Output Validation (Phase 91)
  - Rundeck/Nomad give no actionable error when job is blocked → Dispatch Diagnosis (Phase 88)
  - Rundeck/Airflow/Prefect have no script version history → Script Versioning (Phase 90)
  - All six tools require external tooling for failure alerts → CE Alerting (Phase 89)
- Document the chosen approach for each feature and the rationale

**Section 2 content (RSH-02):**
- Surface: Inline badge below the status column in the job list — no hover or click required
- Queue position shown when reason is capacity-based: "All nodes busy (3rd in queue)"
- Auto-poll: every 5–10 seconds for PENDING jobs currently in view (pick 5 s — consistent with WebSocket live update cadence)
- Coverage: PENDING jobs AND stuck-ASSIGNED jobs
  - Stuck threshold: `timeout_minutes * 1.2`, fallback 30 min when `timeout_minutes` is null
  - Stuck badge text: "Assigned to {node_id} — no completion signal in {N} min"
- Endpoint gap: `GET /jobs/{guid}/dispatch-diagnosis` currently returns 400 for non-PENDING jobs — must be extended to detect stuck-ASSIGNED state server-side (compare `started_at + threshold` to `now()`)
- Client-side vs server-side for stuck detection: server-side — the endpoint computes it, not the frontend timer

**Section 3 content (RSH-03):**
- Chosen mechanism: single outbound webhook URL (HTTP POST)
  - Rationale: works with Slack, Teams, PagerDuty, ntfy.sh, custom endpoints — no SMTP infrastructure required
- Configuration location: Admin → System Config page, new "Notifications" section; stored as `Config` key `alerts.webhook_url`
- CE/EE boundary:
  - CE: single destination URL, fired on `job.failed` events
  - EE: multiple destinations, per-job routing rules, richer payload (stdout, stderr, script content)
- Webhook payload schema:
  ```json
  {
    "event": "job.failed",
    "job_guid": "...",
    "job_name": "nightly-backup",
    "node_id": "node-alpha",
    "error_summary": "exit code 1: permission denied",
    "failed_at": "2026-03-29T22:15:00Z"
  }
  ```
- Implementation note: `webhook_service.py` is currently a CE no-op stub — Phase 89 replaces `dispatch_event()` with real HTTP POST

**Section 4 content (RSH-04):**
- Two separate history tables (separates "what the job does" from "how it's scheduled"):

  Table 1: `job_script_versions` (immutable script snapshots)
  ```
  id               UUID PK
  scheduled_job_id FK → scheduled_jobs.id
  version_number   INT (monotonic per job, starts at 1)
  script_content   TEXT
  signature_id     STR
  signature_payload TEXT
  created_at       DATETIME
  created_by       STR
  ```

  Table 2: `job_definition_history` (metadata audit log)
  ```
  id               UUID PK
  scheduled_job_id FK → scheduled_jobs.id
  changed_at       DATETIME
  changed_by       STR
  diff             TEXT (JSON: {"field": {"from": old_val, "to": new_val}})
  ```

- Linkage: `execution_records` gets a new nullable `script_version_id FK → job_script_versions.id`
  - Existing `script_hash` (SHA-256) is complemented — not replaced — by this FK
- Version trigger: `Config` key `versioning.trigger_mode`
  - `script_changes_only` (default): new version only when `script_content` differs from previous version
  - `any_edit`: every PATCH to the job definition creates a new script version
  - Metadata history (`job_definition_history`) always recorded regardless of toggle
- API endpoints:
  - `GET /jobs/definitions/{id}/versions` — list (version_number, created_at, created_by, first 200 chars of script)
  - `GET /jobs/definitions/{id}/history` — metadata edit log (JSON diffs, chronological)
- Migration: Phase 90 must include `migration_v17.sql` covering both new tables and the new `execution_records` FK column

**Section 5 content (RSH-05):**
- New nullable `validation_rules` JSON column on `scheduled_jobs`:
  ```json
  {
    "exit_code": 0,
    "stdout_regex": "SUCCESS",
    "json_field": {
      "path": "result.status",
      "expected": "ok"
    }
  }
  ```
  All three fields optional; any combination valid; null = no validation = current behaviour preserved
- How scripts signal JSON results: print JSON to stdout — zero node-side changes
  - Node already captures stdout into `ExecutionRecord.stdout`
  - Backend parses stdout as JSON when `json_field` rule is configured
  - Scripts not printing JSON are only subject to `exit_code` and `stdout_regex` checks
- JSON path syntax: dot notation (`result.status`) — simpler than full JSONPath
- New `failure_reason` nullable field on `ExecutionRecord`:
  - `null` — no validation applied or not yet evaluated (current behaviour)
  - `execution_error` — node reported failure, no validation rule violated
  - `validation_exit_code` — `exit_code` rule violated
  - `validation_regex` — `stdout_regex` pattern did not match
  - `validation_json_field` — JSON path/value assertion failed (or stdout not valid JSON)
- Evaluation location: `job_service.py` result processing path (where `ExecutionRecord` is written), not the node
- UI visibility:
  - Execution history list: "Validation failed: stdout_regex" tag instead of generic error badge
  - Job detail panel: which validation rule was violated shown in result section, distinct from runtime errors
