    **Page structure:**
    1. H1 title + 1-2 sentence orientation (Foundry builds custom node images; failures appear in the Templates view or orchestrator logs)
    2. `## Quick Reference` — table with Symptom and Section columns, anchor per H3
    3. Three H2 clusters: `## Build Failures`, `## Smelt-Check Failures`, `## Registry Issues`
    4. H3 per symptom: root cause (2 sentences) → numbered recovery steps → Verify step → escalation

    **Cluster: Build Failures** — H3s:

    1. `Template build status shows FAILED` — root cause: The Docker build process returned a non-zero exit code — most commonly a pip package that does not exist, a base image that could not be pulled, or a Dockerfile syntax error generated by the template engine. The orchestrator captures the last 250 characters of the build output in the template status. Recovery: (1) Check orchestrator logs for `❌ Build Failed:` followed by the truncated Docker output. (2) Identify the failure: "no such package" → check package name in blueprint; "image not found" → verify base image is available and registry is reachable; syntax error → check blueprint JSON for malformed entries. (3) Fix the blueprint and trigger a rebuild. Cross-link to blueprint format: `../feature-guides/foundry.md#blueprints`.

    2. `Template build rejected with 403 — unapproved ingredients` — root cause: The Smelter is running in STRICT enforcement mode and the blueprint contains one or more packages not present in the approved ingredient list. The response body is `Build rejected: Blueprint contains unapproved ingredients: [...]`. Recovery: (1) Review the listed packages. (2) Either submit an approval request for the missing packages (Foundry → Smelter → Add Ingredient) or remove the unapproved package from the blueprint. (3) In WARNING mode, the build proceeds but the template is marked non-compliant. Cross-link: `../feature-guides/foundry.md#smelter`.

    3. `Template build rejected with 403 — ingredient not yet mirrored` — root cause: The package is approved in the Smelter but has not yet been mirrored to the local PyPI or APT mirror (`mirror_status` is not MIRRORED). This prevents builds in air-gapped environments where the node containers cannot reach the public internet. Recovery: wait for the mirror sync job to complete, or manually upload the package via the Smelter upload endpoint. The exact HTTP 403 body is: `Build rejected: Ingredient '<pkg_name>' is approved but not yet mirrored (Status: <status>). Wait for mirroring to complete or upload the package manually.` Cross-link: `../security/air-gap.md`.

    4. `Build log shows warning: environment_service not found` — root cause: The Foundry service constructs the Docker build context by copying `puppets/environment_service/` from the orchestrator container's filesystem. If the path is not found (misconfigured volume mount or wrong working directory), the COPY step in the generated Dockerfile will fail. The warning `⚠️ environment_service not found at <path> — COPY may fail` appears in orchestrator logs before the build starts. Recovery: confirm the orchestrator container has access to the puppets source directory. In production Docker Compose, this directory is baked into the image. If running a custom deployment, verify the volume mounts match the expected path. Tip admonition: "This warning almost always indicates a non-standard deployment path. Standard Docker Compose deployments do not encounter it."

    **Cluster: Smelt-Check Failures** — H3s:

    1. `Template status is STAGING after a successful build` — root cause: The Smelt-Check smoke test (`python --version && pip --version`) ran inside the newly built image and returned a non-zero exit code — the image was built but the Python environment is not functional. The log line is `❌ Smelt-Check FAILED for <template_name>`. Recovery: (1) Check orchestrator logs for the Smelt-Check failure. (2) The most common cause is a base image incompatibility (Alpine vs Debian package naming). Verify the `base_os` field in the blueprint matches the base image OS family. (3) Rebuild the template after fixing the blueprint. STAGING images cannot be deployed to nodes. Cross-link: `../feature-guides/foundry.md#image-lifecycle`.

    2. `Template shows amber warning badge in Templates view` — root cause: The Smelter is in WARNING mode and detected unapproved or unmirrored ingredients. The build succeeded and the Smelt-Check passed, but the template is marked `is_compliant=False`. This is informational — the template can be deployed, but it does not meet the approval policy. Recovery: if compliance is required, switch the Smelter to STRICT mode for future builds, then rebuild after approving the flagged packages. Cross-link: `../feature-guides/foundry.md#smelter`.

    **Cluster: Registry Issues** — H3s:

    1. `Template shows a last built time but image is not available on nodes` — root cause: The Docker build succeeded but the subsequent `docker push` to the local registry (`localhost:5000`) failed silently. The template status field starts with `PUSH_FAILED:`. The orchestrator does not raise an error — it logs the failure and continues. Recovery: (1) Check orchestrator logs for `PUSH_FAILED`. (2) Confirm the local registry container is running: `docker ps | grep registry`. (3) If registry is down, start it and re-trigger the template build. Warning admonition: "The dashboard may show 'last built at <time>' even when the push failed — always check orchestrator logs if nodes cannot pull the image."

    2. `Node container fails to pull image from the local registry at startup` — root cause: The node container cannot reach `localhost:5000` from inside the Docker bridge network. `localhost` inside a container resolves to the container itself, not the host. Recovery: use the Docker host's bridge IP (`172.17.0.1`) or the hostname `host.docker.internal` in the image URL when configuring node containers. Set `extra_hosts: host.docker.internal:172.17.0.1` in the node compose file. Tip: the validation node compose files in `mop_validation/local_nodes/` use the correct networking pattern and can be used as a reference.

    **Formatting rules:** Same as nodes.md and jobs.md. No collapsible blocks. Admonitions outside numbered lists.

    **Required entries (from locked CONTEXT.md decisions) — write all four:**

    **Entry 1: Blueprint packages must use `{"python": [...]}` dict format**
    H3: "Blueprint packages are not installed during Foundry builds"
    Answer: Blueprint package lists must be structured as a dict with a `"python"` key, not a plain list. The Foundry service calls `packages.get("python", [])` — a plain list returns an empty result, silently skipping all installations. The build succeeds but the node image has none of the expected packages.
    Code block:
    ```json
    // Correct
    {"python": ["requests", "numpy"]}

    // Wrong — silently skips all packages
    ["requests", "numpy"]
    ```
    Cross-link: `../feature-guides/foundry.md#blueprints`

    **Entry 2: EXECUTION_MODE=direct for Docker-in-Docker nodes**
    H3: "Node container fails to run jobs — RuntimeError: No container runtime found"
    Answer: When a puppet node runs inside a Docker container (e.g., a Foundry-built image deployed via Docker Compose), EXECUTION_MODE must be set to `direct`. In `auto` mode, the node tries to spawn Podman or Docker sub-containers. Inside an existing Docker container, this fails due to cgroup v2 conflicts. `direct` mode runs scripts in the node process directly without a container wrapper.
    Code block:
    ```yaml
    # In node-compose.yaml environment section
    EXECUTION_MODE: direct
    ```
    Cross-link: Foundry runbook `foundry.md#build-failures` if the node image was Foundry-built.

    **Entry 3: JOIN_TOKEN is base64-encoded JSON — not a plain API key**
    H3: "Node fails to enroll — Token payload missing 't' or 'ca'"
    Answer: The JOIN_TOKEN is a base64-encoded JSON object containing two keys: `t` (the actual token) and `ca` (the Root CA PEM). It is not a plain string or a JWT. Even a single trailing newline or URL-encoding artefact in the token value will cause the `Token payload missing 't' or 'ca'` error and abort enrollment. Copy the token exactly as generated from Admin → Nodes → Generate Join Token.
    Code block (show what it decodes to, not the actual token):
    ```json
    // JOIN_TOKEN decodes to this structure (never construct manually):
    {
      "t": "<actual_token_string>",
      "ca": "-----BEGIN CERTIFICATE-----\n...\n-----END CERTIFICATE-----\n"
    }
    ```
    Cross-link: `../security/mtls.md#the-join_token`

    **Entry 4: ADMIN_PASSWORD in .env only seeds on first startup**
    H3: "Changing ADMIN_PASSWORD in .env has no effect"
    Answer: The `ADMIN_PASSWORD` environment variable is read only once — when the orchestrator starts and the `admin` user does not yet exist in the database. Once the admin account exists, the env var is ignored entirely. The database password is the source of truth for existing deployments. To change the admin password after first run, use Admin → Users → admin → Reset Password in the dashboard.
    Warning admonition: "Do not drop the database to reset the password — this destroys all nodes, jobs, and templates. Use the dashboard Reset Password flow instead."
    Code block: show the startup logic comment: "# Only seeds if admin user does not exist"

    **Required how-to entries (write all three):**

    **Entry 5: How do I reset a node without re-enrolling?**
    H3: "How do I reset a node identity without re-enrolling?"
    Answer: You cannot reset a node's identity without re-enrolling. The node identity is bound to its client certificate. To assign a new identity: stop the container, delete `secrets/node-*.crt` and `secrets/node-*.key` from the secrets volume, and restart with a fresh JOIN_TOKEN. The orchestrator registers it as a new node. The old Offline entry can be deleted from the Nodes view.
    Cross-link: `../security/mtls.md#certificate-rotation`

    **Entry 6: Scheduled jobs not running at expected time (UTC)**
    H3: "My scheduled job does not run at the expected local time"
    Answer: APScheduler (the orchestrator's scheduling engine) evaluates all cron expressions in UTC. There is no per-job timezone configuration. A cron expression `0 9 * * *` fires at 09:00 UTC regardless of the server's local timezone. Convert your intended local time to UTC before entering the cron field.
    Tip admonition: "Use a UTC cron calculator to verify your expression. For example, 09:00 BST (UTC+1) = `0 8 * * *`."
    Cross-link: `../feature-guides/job-scheduling.md`

    **Entry 7: Can I run jobs without Ed25519 signing?**
    H3: "Can I submit jobs without Ed25519 signing?"
    Answer: No. Signature verification is enforced at the node before any script is executed — it is not configurable. A job submitted without a valid Ed25519 signature is immediately rejected with status `SECURITY_REJECTED`. This enforcement is by design: the node must cryptographically confirm the script came from an authorised operator before running it.
    Danger admonition: "There is no configuration flag, environment variable, or API option to disable signature verification. This is a security invariant."
    Cross-link: `../feature-guides/mop-push.md#ed25519-key-setup`

    **Additional recommended entries (write at least 3 of these 4):**

    **Entry 8: Node appears multiple times in dashboard**
    H3: "Why does my node appear multiple times in the Nodes view?"
    Answer: Duplicate entries happen when the `secrets/` volume is recreated or the `node-*.crt` files are deleted. On restart, the node generates a new UUID and enrolls fresh, creating a second DB entry. The old entry remains Offline. Recovery: delete the Offline duplicate from the Nodes view. No data loss occurs.

    **Entry 9: Foundry build stuck in STAGING**
    H3: "Why is my Foundry template stuck in STAGING status?"
    Answer: A template in STAGING status means the Docker build succeeded but the post-build Smelt-Check smoke test failed. The image was built but the Python environment inside it is not functional. Check orchestrator logs for `❌ Smelt-Check FAILED for <template_name>`, identify the cause (usually a base OS / package manager mismatch), fix the blueprint, and rebuild.
    Cross-link: `foundry.md#smelt-check-failures`

    **Entry 10: Node keeps getting TAMPERED status**
    H3: "Why does my node keep showing TAMPERED status?"
    Answer: The zero-trust capability guard detected that the node reported a capability not present in its registered `expected_capabilities`. This happens when a tool is installed inside the container after the template was built, or when capability detection logic changes between node restarts. Investigate via `docker exec <node-container> <capability_check_command>`, clear the flag via Nodes → [node] → Clear Tamper, and rebuild the template if the capability is legitimate.

    - Code blocks use appropriate language tags: `json`, `yaml`, `bash`
    - Warning/danger admonitions for irreversible actions (dropping DB, revocation)
    - Tip admonitions for shortcuts
    - End each entry with a cross-link line where a deeper guide exists
