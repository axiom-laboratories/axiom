  <action>
    Replace the stub at docs/docs/runbooks/jobs.md with the full runbook. Follow the locked structure from 25-CONTEXT.md:

    **Page structure:**
    1. H1 title + 1-2 sentence orientation
    2. `## Quick Reference` — table with "Symptom" and "Section" columns, anchor link per symptom
    3. Three H2 clusters: `## Dispatch Failures`, `## Signing Errors`, `## Timeout Patterns`
    4. H3 per symptom; each H3 follows: root cause (2 sentences) → numbered recovery steps → Verify step (command + expected output) → "If the issue persists" escalation

    **Cluster: Dispatch Failures** — H3s for each:

    1. `Job stuck in Queued / PENDING indefinitely` — root cause: The orchestrator found no node with matching tags, required capabilities, and sufficient memory budget. Jobs remain PENDING until a matching node checks in. Recovery steps: (1) Check the job's `capability_requirements` and `target_tags` in the Jobs view. (2) Confirm at least one Online node has those capabilities (Nodes view → select node → capabilities). (3) If no node qualifies, reduce requirements or enroll a node with the needed capabilities. (4) Check node memory budgets: if the job's `memory_limit` is set, nodes with insufficient headroom are skipped.
       Verify: confirm job status changes from PENDING to ASSIGNED after a matching node checks in.
       Escalation: if still PENDING after a node with matching capabilities is Online, check orchestrator logs for node selection loop output.

    2. `Job status is BLOCKED` — root cause: The job has a `depends_on` dependency that has not yet reached COMPLETED status. BLOCKED is expected behaviour — the job will unblock automatically when the dependency completes. Recovery: check the dependency job's status in the Jobs view. If the dependency is FAILED or DEAD_LETTER, the dependent job will be CANCELLED (see next). No action needed if waiting.
       Verify: observe the dependency job completing; the dependent job moves to QUEUED automatically.

    3. `Job status is CANCELLED with "cancelled because upstream failed"` — root cause: The upstream dependency job failed, and the orchestrator propagated cancellation to all downstream dependents. The log line `🚫 Job <guid> cancelled because upstream <guid> failed` appears in orchestrator logs. Recovery: investigate the upstream job's failure (re-run or fix the upstream), then resubmit the cancelled job.
       Include the exact log line in a code block.

    4. `Job status is DEAD_LETTER` — root cause: The job failed on every retry attempt (up to `max_retries`). The orchestrator marks it DEAD_LETTER and creates an alert. Recovery: check the job's execution output for the error message. Fix the underlying script issue, then resubmit. The DEAD_LETTER job itself cannot be retried — submit a new job.
       Include the exact log line `Job <guid> exhausted all <N> retries and failed terminally.` in a code block.

    5. `Job status is ZOMBIE_REAPED` — root cause: The job was ASSIGNED to a node but the node stopped sending execution updates (crash, OOM kill, or network partition). The zombie reaper reclaims the job after `zombie_timeout_minutes` (default: 30 min). Recovery: check if the assigned node is still Online. If the node OOM crashed, increase its `job_memory_limit` or reduce the job's `memory_limit`. The reaped job returns to QUEUED for reassignment (if retries remain) or to FAILED (if retries exhausted).

    **Cluster: Signing Errors** — H3s for each:

    1. `Job status is SECURITY_REJECTED — Signature Verification FAILED` — root cause: The node verified the job's signature against the registered public key and the check failed — either the script was signed with a different Ed25519 key than the one registered in Signatures, or the script content was modified after signing. The exact node log line is `❌ Signature Verification FAILED for Job <guid>: <exception detail>`. Recovery: (1) Check which key was used to sign (mop-push shows key fingerprint). (2) Confirm the matching public key is registered in the dashboard Signatures view. (3) Re-sign the script with the correct key and resubmit. Cross-link: `../feature-guides/mop-push.md#ed25519-key-setup`.

    2. `Job status is SECURITY_REJECTED — Verification Key missing` — root cause: The node could not fetch the verification public key from the orchestrator's `/verification-key` endpoint at startup — usually a transient connectivity issue or the node started before the orchestrator was ready. The exact log line is `❌ CRITICAL: Verification Key missing. Cannot verify signature.`. Recovery: restart the node container. On restart, the node re-fetches the verification key. If the error persists, confirm orchestrator is reachable at AGENT_URL from the node network.

    3. `Job status is SECURITY_REJECTED — Missing script or signature` — root cause: The job payload was submitted without a `signature` field. This happens when a job is submitted via the API directly (not via mop-push) or when a script was edited after signing. The node returns `Missing script or signature` in the execution result. Recovery: resubmit the job using mop-push, which handles signing automatically. Danger admonition: "Bypassing the signing workflow is not supported. Signature verification is enforced at the node before execution — there is no configuration flag to disable it." Cross-link: `../feature-guides/mop-push.md#ed25519-key-setup`.

    **Cluster: Timeout Patterns** — H3s for each:

    1. `Job moves from ASSIGNED to FAILED or ZOMBIE_REAPED after ~30 minutes` — root cause: The zombie reaper fires at `zombie_timeout_minutes` (default: 30 minutes, configurable via Admin → Configuration in the dashboard). If the executing node stopped reporting progress within this window, the orchestrator reclaims the job. Recovery: (1) If the job genuinely needs more than 30 minutes, increase `zombie_timeout_minutes` via Admin → Configuration. (2) If the node crashed mid-execution, investigate with `docker logs <node-container>` for OOM or runtime errors.
       Note: this is the effective operator-visible timeout. The 30-minute window is configurable.

    2. `Node never reports job result — job stays ASSIGNED indefinitely` — root cause: The node container was OOM killed (memory_limit hit) or suffered a network partition. The heartbeat stops but the zombie reaper has not yet fired. Recovery: check node status in the dashboard — an OOM node usually goes Offline. Check `docker stats <node-container>` for memory usage near the limit. Increase `job_memory_limit` on the node template or reduce the job's `memory_limit`. The zombie reaper will reclaim the job within `zombie_timeout_minutes`.

    **Formatting rules (same as nodes.md):**
    - Backticks for all log lines, status values, env vars, file paths, CLI commands
    - `!!! warning` for security-adjacent notes (signing enforcement, no bypass)
    - `!!! danger` for irreversible states (DEAD_LETTER cannot be retried — must resubmit)
    - `!!! tip` for shortcuts
    - No collapsible `???` blocks
    - Admonitions before/after numbered lists, never inside them
    - Jump table uses plain-text anchor-friendly symptom descriptions
