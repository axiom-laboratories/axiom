    2. `## Quick Reference` — Markdown table with two columns: "Symptom" and "Section" — one row per H3 symptom section, each with an anchor link
    3. Three H2 cluster sections: `## Enrollment Failures`, `## Heartbeat Loss`, `## Certificate Errors`
    4. Within each cluster: H3 headers for individual symptoms

    **Per-symptom structure (mandatory for every H3):**
    - 2-sentence root cause (why this happens, mechanically)
    - `**Recovery steps:**` with numbered list
    - `**Verify it worked:**` with a fenced bash code block and "Expected output:" line
    - "If the issue persists after these steps, [action]."

    **Cluster: Enrollment Failures** — cover these symptoms as H3s:
    1. `Node container exits immediately at startup` — root cause: JOIN_TOKEN malformed, AGENT_URL unreachable, or stale cert files from a previous enrollment. Recovery: check docker logs for the exact error line, verify AGENT_URL is reachable, check if secrets/ volume has old node-*.crt files. If stale certs: stop container, delete secrets/node-*.crt and secrets/node-*.key, restart.
    2. `Enrollment Failed: [Errno 111] Connection refused` — root cause: AGENT_URL points to wrong host or port. Recovery: verify AGENT_URL env var matches orchestrator address and port (default 8001), confirm orchestrator container is running, confirm network connectivity from node container.
    3. `Enrollment Failed: SSL: CERTIFICATE_VERIFY_FAILED` — root cause: The Root CA PEM embedded in JOIN_TOKEN does not match the TLS certificate the orchestrator is presenting. Recovery: regenerate the JOIN_TOKEN from the dashboard (Admin → Nodes → Generate Join Token), ensuring the token was generated against the current orchestrator TLS setup. Cross-link to `../security/mtls.md#the-join_token`.
    4. `Token payload missing 't' or 'ca'` — root cause: JOIN_TOKEN is a legacy plain token, not an Enhanced Token (base64-encoded JSON with `t` and `ca` keys). Recovery: regenerate from dashboard; the Enhanced Token is the only supported format. Add warning admonition: "Do not truncate, URL-encode, or add whitespace to the token value — even a trailing newline causes this error."
    5. `Node appears as a duplicate entry in the Nodes view` — root cause: secrets/ volume was recreated or the node-*.crt files were deleted — _load_or_generate_node_id() generated a new UUID and the node enrolled fresh while the old entry remained. Recovery: delete the old Offline entry from the Nodes view in the dashboard; the current node is the newer one. Cross-link to FAQ: `faq.md#why-does-my-node-appear-multiple-times-in-the-dashboard`.

    **Cluster: Heartbeat Loss** — cover these symptoms as H3s:
    1. `Node goes Offline in dashboard after container restart` — root cause: The heartbeat thread starts before cert files are available (cert files may be in a slow-mounting volume). The thread loops waiting for cert files; if not found, heartbeat never reports. Recovery: verify the secrets/ volume is mounted correctly and node-*.crt exists; check docker logs for `[Heartbeat] Failed`. If cert files are present but corrupt, re-enroll.
    2. `Node status stays Offline after cert rotation` — root cause: The old revoked certificate is still in the secrets/ volume. _load_or_generate_node_id() picks up the old file, the node attempts to use the revoked cert, and /work/pull returns 403. Recovery: stop the node container, delete secrets/node-*.crt and secrets/node-*.key, restart with a new JOIN_TOKEN. Cross-link to `../security/mtls.md#certificate-rotation`.
    3. `Node status shows TAMPERED in dashboard` — root cause: The zero-trust capability guard in the orchestrator detected that the node reported a capability not present in expected_capabilities at enrollment time — this indicates the container environment changed after the image was built. Recovery: investigate the node's current environment (docker exec into the node container, run the capability check manually), then either clear the tamper flag via Nodes → [node] → Clear Tamper or rebuild the template with the updated capability set.

    **Cluster: Certificate Errors** — cover these symptoms as H3s:
    1. `Node gets HTTP 403 on /work/pull and stops receiving jobs` — root cause: The node's client certificate has been revoked in the orchestrator's RevokedCert table. The /work/pull endpoint checks cert serial against this table and returns 403. Recovery: this is intentional — a revoked node cannot receive jobs. If the revocation was accidental, it cannot be undone (revocation is permanent). Re-enroll the node with a fresh JOIN_TOKEN and new cert. Warning admonition: "Cert revocation cannot be undone. Before revoking, confirm the node identity in the Nodes view." Cross-link to `../security/mtls.md#certificate-revocation`.
    2. `Node receives concurrency_limit: 0 and processes no jobs` — root cause: The orchestrator sends concurrency=0 when a node's template is in REVOKED lifecycle state or when the node has TAMPERED status — this halts job dispatch without a hard 403. Recovery: check the node's template lifecycle state in the Foundry view (Templates tab); if REVOKED, rebuild or assign the node a different template. If TAMPERED, resolve via the heartbeat loss cluster above.

    **Formatting rules to follow:**
    - Use backticks for all log lines, env var names, file paths, and CLI commands
    - Use `!!! warning` admonitions for irreversible operations (revocation permanence, cert deletion)
    - Use `!!! tip` for shortcuts or discovery hints
    - No collapsible `???` blocks
    - Admonitions go BEFORE or AFTER numbered lists, never inside them (avoids MkDocs list counter reset)
    - Jump table anchors: use plain text H3 headers (no backtick-wrapped log lines as H3 anchor targets in the jump table — use readable symptom descriptions). The H3 content can start with a code snippet but the H3 header itself must be plain text for reliable anchor generation.
    - Verify step for each symptom: use `docker logs <container-name>` or `docker ps` as the verification command — these are universally available to operators
