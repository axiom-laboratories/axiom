# Plan 94-02: Write Competitor Product Notes File

## Objective

Create `mop_validation/reports/competitor_product_notes.md` — a self-referencing product notes document
that distills actionable observations from the existing competitor pain-points report. The file is
aimed at informing MoP product and messaging decisions, not duplicating the source report.

## Context

- Source report: `mop_validation/reports/plans/20262903/competitor_pain_points.md` — comprehensive analysis
  of Rundeck, AWX, Nomad+Vault, Temporal, Airflow, Prefect (553 lines, 6 tools, 5 cross-cutting themes)
- The five cross-cutting pain categories are: upgrade trauma, ACL/secrets complexity, observability gaps,
  false infrastructure abstraction, conceptual surface area
- Primary lens for this notes file: **gaps MoP should close** — where competitor pain maps to current MoP
  weaknesses or unclaimed positioning opportunities
- The "Implications for Your Tool" section at the end of the source report is the strongest starting point
- Format: self-referencing (brief intro, link to source, observations) — not a copy of the source
- Each observation tagged: [Positioning] / [Feature] / [Messaging]
- Target: 5–7 observations covering themes from across all 6 competitors

<task id="94-02-T2">
**Draft 5–7 actionable observations**

Each observation must:
- Name a specific pain from the competitor report (with tool attribution)
- State what MoP currently does or should do differently
- Be tagged: [Positioning] (we're already better), [Feature] (we should build this), or [Messaging] (we should say this)
- Be 2–4 sentences — tight, not padded

**Theme guidance from source report:**

1. **Upgrade trauma** (Rundeck, AWX, Airflow, Prefect — all High): MoP uses `create_all` with manual migration SQL — the upgrade path today requires manual ALTER TABLE steps (migration_v*.sql). This is better than AWX (data loss risk) but not yet frictionless. [Feature] — consider migration tooling or at minimum a single migration script that handles all versions.

2. **Error messages and debuggability** (Universal): MoP's dispatch diagnosis UI (Phase 88) already addresses the "why is my job pending?" gap that competitors leave blank. This is a differentiator. [Positioning] + [Messaging].

3. **Observability** (Airflow, Temporal — High): MoP's output validation (Phase 91) directly addresses the "task ran but was data correct?" gap. [Positioning] — use this in messaging against Airflow and Temporal.

4. **Access control opacity** (Rundeck — High): MoP's RBAC gives actionable permission denials (HTTP 403 with clear role/permission logged in audit trail). Rundeck's ACL failure mode is silent. [Positioning] + [Messaging].

5. **Local dev friction** (AWX requires Kubernetes, Airflow excludes Windows, Temporal requires 3 services): MoP runs as a single compose stack on Linux/Mac/Windows. The Windows getting-started path (PR #13) specifically closes the Windows local dev gap Airflow has. [Positioning] + [Messaging].

6. **Conceptual surface area** (Nomad+Vault, Temporal — High): MoP's model is: nodes, jobs, schedules. No "workflows", "tasks", "DAGs", "agents vs workers", no separate secret store. Lower conceptual overhead is a genuine on-ramp advantage. [Messaging].

7. **Project/vendor health risk** (AWX existential uncertainty, Prefect API instability): MoP is MIT-licensed CE with a clearly separated EE tier. No vendor lock-in, no proprietary binary-only core. [Positioning] + [Messaging] for operators evaluating long-term commitments.
</task>
