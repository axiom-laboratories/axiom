## Workflow CRUD API Status

The Phase 146 implementation (completed 2026-04-15) provides:
- `POST /api/workflows` — create with DAG validation
- `GET /api/workflows` — list all (metadata only, no nested graph)
- `GET /api/workflows/{id}` — fetch single workflow with full graph
- `PUT /api/workflows/{id}` — atomic replace of steps/edges/parameters
- `DELETE /api/workflows/{id}` — delete (blocked if active WorkflowRuns exist)
- `POST /api/workflows/{id}/fork` — clone workflow and pause source
- `POST /api/workflows/validate` — validate DAG without saving

WorkflowService layer provides:
- `validate_dag()` — cycle detection via networkx, referential integrity checks, depth limit (30 levels)
- `create()`, `list()`, `get()`, `update()`, `delete()`, `fork()` — CRUD methods with nested transaction handling
- Error payloads: `INVALID_EDGE_REFERENCE`, `CYCLE_DETECTED`, `DEPTH_LIMIT_EXCEEDED`, `ACTIVE_RUNS_EXIST`

conftest.py fixtures already provide:
- `async_client` — FastAPI TestClient via httpx.AsyncClient
- `auth_headers` — JWT token for admin user
- `async_db_session` — transactional SQLite for test isolation
- `workflow_fixture` — pre-created workflow with 3-step linear chain (Step 1 → Step 2 → Step 3)
- `workflow_run_fixture` — WorkflowRun in RUNNING state (Phase 147)

Mapped from Phase 146 API routes:
- HTTP 422: `{"error": "CYCLE_DETECTED", "cycle_path": [...]}`

- HTTP 409: `{"error": "ACTIVE_RUNS_EXIST", "active_run_ids": [...]}`
- HTTP 200/201: successful response with full workflow object
