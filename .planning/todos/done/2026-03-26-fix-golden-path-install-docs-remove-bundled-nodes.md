## Problem

The GHCR install method (`compose.cold-start.yaml`) currently mentions that two puppet nodes start automatically, requiring `JOIN_TOKEN_1` and `JOIN_TOKEN_2` in `.env`. This is the wrong golden path — it conflates orchestrator setup with node deployment, which belong to separate steps.

The correct golden path should be:
1. Docker Compose to pull and start the control plane (dashboard, orchestrator/agent service, PostgreSQL, Caddy)
2. Get the orchestrator up and running (set `.env`, start stack, log in)
3. Deploy a node **via the GUI or the enroll script provided by the orchestrator** — not as part of the initial compose
4. Node image pulled from CE hosted registry (GHCR) or EE self-hosted registry

## Solution

- Remove `JOIN_TOKEN_1` / `JOIN_TOKEN_2` from the install doc `.env` example
- Remove or rewrite the line stating "the two built-in puppet nodes start automatically" (`install.md:97`)
- `compose.cold-start.yaml` (or its successor) should only define control-plane services (caddy, agent, postgres) — no node services
- Enroll-node doc should be the canonical place where a node is first deployed, using the token generated by the running orchestrator
- Consider renaming `compose.cold-start.yaml` → `compose.server.yaml` (or aligning with the existing name) to avoid confusion
