## Summary

v22.0 Security Hardening addresses container security posture and EE licence protection across all stack components. Container hardening removes privileged mode, drops unnecessary capabilities, restricts network access, and enforces non-root users. EE licence protection adds signed wheel manifest verification, HMAC-keyed boot logs, entry point whitelisting, and release-time signing tools.

**Requirements:** 16 total
- **Container Hardening (CONT-01 to CONT-10):** 10 requirements
- **EE Licence Protection (EE-01 to EE-06):** 6 requirements

**Coverage:** 16/16 requirements mapped (100%)

## Phase Details

### Phase 132: Non-Root User Foundation

**Goal:** All containers run as non-root appuser (UID 1000) with correct file ownership on volumes and across upgrades.

**Depends on:** Nothing (foundation phase)

**Requirements:** CONT-01, CONT-06

**Success Criteria** (what must be TRUE):
1. Container processes show uid 1000 in ps output and /proc
2. App directories and mounted volumes owned by uid 1000
3. Volume ownership migrates correctly when upgrading from root containers
4. Non-root user can write to necessary directories (logs, tmp, config)

**Plans:** 
- [x] **Plan 01:** Update Containerfile.server and Containerfile.node with appuser + chown + USER (COMPLETE — 2026-04-12)
- [ ] **Plan 02:** Rebuild images and validate via docker compose + test suite

### Phase 133: Network & Security Capabilities

**Goal:** Drop all unnecessary Linux capabilities, disable privilege escalation, and restrict Postgres to loopback-only access.

**Depends on:** Phase 132

**Requirements:** CONT-03, CONT-04

**Success Criteria** (what must be TRUE):
1. Postgres accessible only on 127.0.0.1:5432 (external bind fails or is unavailable)
2. All compose services have `cap_drop: ALL` + `security_opt: no-new-privileges` set
3. Caddy has `cap_add: NET_BIND_SERVICE` to allow binding to port 80/443
4. No security warnings from `docker inspect` or `podman inspect` regarding dropped caps

### Phase 134: Socket Mount & Podman Support

**Goal:** Remove `privileged: true` from node containers and use Docker/Podman socket mount instead; auto-detect Podman socket.

**Depends on:** Phase 132

**Requirements:** CONT-02, CONT-09, CONT-10

**Success Criteria** (what must be TRUE):
1. Node container removes `privileged: true` and mounts `/var/run/docker.sock` or `/run/podman/podman.sock`
2. Node can execute jobs without privileged mode (container-in-container works via socket)
3. `runtime.py` auto-detects Podman socket at `/run/podman/podman.sock` and uses it
4. Podman-variant compose file (`node-compose.podman.yaml`) ships and works with local Podman host

### Phase 137: Signed EE Wheel Manifest

**Goal:** Verify Ed25519-signed wheel manifest (containing SHA256 hash) before installing EE wheel.

**Depends on:** Nothing (independent)

**Requirements:** EE-01

**Success Criteria** (what must be TRUE):
1. EE wheel installation reads signed manifest and verifies signature
2. Installation fails with RuntimeError if signature verification fails
3. Installation fails if wheel SHA256 does not match manifest
4. Valid, correctly-signed wheels install successfully

### Phase 138: HMAC-Keyed Boot Log

**Goal:** Replace plain SHA256 hash chain with HMAC-SHA256 boot log; accept legacy SHA256 entries on read for backward compatibility.

**Depends on:** Nothing (independent)

**Requirements:** EE-02, EE-03

**Success Criteria** (what must be TRUE):
1. Boot log entries use HMAC-SHA256 keyed on ENCRYPTION_KEY
2. New boot log entries computed with HMAC verify correctly
3. Legacy SHA256 chain entries are still accepted when reading boot log
4. No forced migration on upgrade — mixed SHA256/HMAC entries coexist

**Plans:** TBD

### Phase 139: Entry Point Whitelist & Enforcement

**Goal:** Whitelist importlib entry points to only load `ee.plugin:EEPlugin`; enforce ENCRYPTION_KEY presence with hard RuntimeError.

**Depends on:** Nothing (independent)

**Requirements:** EE-04, EE-06

**Success Criteria** (what must be TRUE):
1. Entry point validation checks `ep.value == "ee.plugin:EEPlugin"` before loading
2. Untrusted entry points raise RuntimeError and prevent startup
3. ENCRYPTION_KEY absence in production raises hard RuntimeError (no fallback)
4. CE mode can start without ENCRYPTION_KEY (if applicable)

## Architecture Notes

**Container Hardening Phases (132–136):**
- Foundation first (Phase 132: users/ownership)
- Then perimeter (Phase 133: network/capabilities)
- Then runtime (Phase 134: socket mount)
- Then optimization (Phase 135: limits/cleanup)
- Then propagation (Phase 136: user in generated images)

**EE Licence Protection Phases (137–140):**
- Largely independent work
- Could be developed in parallel with container hardening
- Phases 137/139 can overlap (wheel manifest + entry point validation)
- Phase 138 (boot log) is independent

**Derivation:**
- All 9 phases derived from 16 requirements
- No arbitrary structure imposed
- Success criteria are observable user/operator behaviors
- No duplicates; 100% coverage maintained
