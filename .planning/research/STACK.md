## Recommended Stack Additions

### Core: GitHub Actions Deployment Chain (3 new actions)

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| `actions/configure-pages` | `v5` | Configures the GH Pages environment and injects the correct base `site_url` for asset path resolution | Required by GitHub's native Pages pipeline; prevents asset 404s when deploying to a subpath (e.g. `/<repo>/`) |
| `actions/upload-pages-artifact` | `v3` | Packages the built `docs/site/` directory as a deployable Pages artifact | GitHub's first-party format for Pages deployments; v3 is the current required minimum after January 2025 deprecation of v2 |
| `actions/deploy-pages` | `v4` | Atomically deploys the uploaded artifact to the `github-pages` environment | First-party; shows deployment status in Actions UI; v4 pairs with upload-pages-artifact v3 — they must be used together |

**Why this chain over `mkdocs gh-deploy --force`:**
`mkdocs gh-deploy` commits directly to a `gh-pages` branch using `ghp-import`. It works, but requires `contents: write` token permission and creates noisy branch-push commits in the history. The `configure-pages` + `upload-pages-artifact` + `deploy-pages` chain uses the GitHub-native Pages environment with `pages: write` + `id-token: write` permissions (OIDC-based, no long-lived tokens), gives deployment status in the Actions UI, and is what the official MkDocs Material documentation now recommends for new setups.

### No New Python Libraries Required

The existing `docs/requirements.txt` (`mkdocs-material==9.7.5` + `mkdocs-swagger-ui-tag==0.8.0`) is sufficient for the GH Pages build. The GitHub Actions workflow installs these exact same packages with `pip install -r docs/requirements.txt` — same as the existing `ci.yml` `docs` job. No additions to `requirements.txt`.

## Configuration Changes Required

### 1. `mkdocs.yml` — `site_url`

**Current value:**
```yaml
site_url: https://dev.master-of-puppets.work/docs/
```

**Required for GitHub Pages (no custom domain):**
```yaml
site_url: https://<org>.github.io/<repo>/
```

**Required for GitHub Pages (custom domain):**
```yaml
site_url: https://docs.axiom.sh/
```

This is the most important config change. MkDocs uses `site_url` to construct all absolute asset URLs. Without the correct value:
- The `privacy` plugin caches external assets under wrong paths, causing 404s for fonts and stylesheets
- `mkdocs-swagger-ui-tag` derives `oauth2RedirectUrl` from `site_url` — if wrong, the Swagger UI OAuth flow 404s
- The `search` plugin builds wrong sitemap URLs
- All internal cross-references using absolute paths break

### 2. `mkdocs.yml` — `offline` plugin

**Current config:**
```yaml
plugins:
  - offline
```

**Problem:** The `offline` plugin forces `use_directory_urls: false` globally, which rewrites all internal link paths to flat filenames (e.g. `getting-started/install/index.html` becomes `getting-started/install.html`). This is correct for distributing a `.zip` archive to be opened from the local filesystem but breaks GitHub Pages navigation where directory URLs are expected and where the web server resolves `/getting-started/install/` to the `index.html` inside that directory.

**Required change — make `offline` conditional:**
```yaml
plugins:
  - search
  - privacy
  - offline:
      enabled: !ENV [OFFLINE, false]
  - swagger-ui-tag
```

The GH Actions workflow does not set `OFFLINE`, so the offline plugin is inactive for the GH Pages build. The container `Dockerfile` builder stage can set `OFFLINE=true` if desired, but the current Dockerfile does not set it — so this change has no effect on the container build. The `privacy` plugin is not affected and works correctly on both hosted and offline builds.

## New Workflow File

**File:** `.github/workflows/docs.yml`

**Trigger:** `push` to `main` only (not on PRs — deploying on every PR would overwrite the live public site with work-in-progress content).

**Required permissions** (job level):
```yaml
permissions:
  pages: write
  id-token: write
```

**Required environment declaration:**
```yaml
environment:
  name: github-pages
  url: ${{ steps.deployment.outputs.page_url }}
```

This links the job to the GH Pages deployment environment created in repository settings, enabling the deployment status badge and protection rules.

**Build steps:** Identical to the existing `ci.yml` `docs` job — `actions/setup-python@v5` + `pip install -r docs/requirements.txt` + `mkdocs build --strict`. No new build logic.

**No FastAPI dependencies in CI:** The `openapi.json` is pre-committed at `docs/docs/api-reference/openapi.json`. The GH Actions workflow runs only `mkdocs build` — it never executes the FastAPI app or `export_openapi.py`. This keeps CI fast and avoids pulling in all of `puppeteer/requirements.txt` (FastAPI, SQLAlchemy, asyncpg, and ~30 other packages) into the docs build.

**Regeneration of `openapi.json`:** A local script (e.g. `scripts/regen-openapi.sh`) handles regenerating `openapi.json` from the live FastAPI app when the API schema changes. The developer runs it locally and commits the updated file. This is a manual step, deliberately kept out of CI.

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|-------------|-------------|-------------------------|
| `upload-pages-artifact` + `deploy-pages` GH-native chain | `mkdocs gh-deploy --force` (ghp-import) | Only if the repo already has an established `gh-pages` branch history that must be preserved; new setups should use the GH-native chain |
| Pre-committed `openapi.json` — no FastAPI in CI | Generate `openapi.json` in CI (install all of `puppeteer/requirements.txt`) | Only if API schema changes very frequently and stale docs are a real problem; adds ~60–90s to CI and introduces the full FastAPI dependency graph into the docs build |
| Separate `docs.yml` workflow | Add a deploy step to the existing `ci.yml` | Avoid — mixing test and deploy concerns in one file makes it harder to trigger deploy independently (e.g. on a docs-only commit) and harder to scope permissions correctly |
| Deploy on push to `main` | Deploy on version tag push | Use tag-based deploy only if the site needs versioning (e.g. `v1/` and `v2/` subdirectories via `mike`); for a single-version CE public site, `main`-branch continuous deploy is correct |
| Conditional `offline` plugin via `!ENV` | Remove `offline` plugin entirely from `mkdocs.yml` | Remove it only if the container build (air-gapped deployment) is no longer a target; keeping it conditional preserves both use cases |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| Third-party `mhausenblas/mkdocs-deploy-gh-pages` Action | Unmaintained community wrapper; adds no capability over the GH-native chain | `actions/upload-pages-artifact@v3` + `actions/deploy-pages@v4` |
| `actions/upload-pages-artifact@v2` or `actions/deploy-pages@v3` | Deprecated as of January 2025; GitHub will reject these | `@v3` and `@v4` respectively — must be used as a matched pair |
| `mkdocs serve` in CI | Not production-safe (MkDocs upstream issue #1825); not needed for static site deploy | `mkdocs build --strict` only |
| Setting `OFFLINE=true` in the GH Actions deploy environment | Forces `use_directory_urls: false`, which breaks directory-URL navigation on GH Pages | Omit `OFFLINE` env var entirely in the deploy workflow |
| Hardcoding the current `site_url` (`https://dev.master-of-puppets.work/docs/`) in the GH Pages workflow | Assets and cross-references will resolve against the internal Caddy hostname, not the GH Pages domain | Set `site_url` in `mkdocs.yml` to the GH Pages URL, or override via `--config-file` / env in the workflow |
| `mike` for versioned docs | Adds significant complexity (version switcher, alias management, separate branch structure); not needed for single-version CE public documentation | Plain `deploy-pages` |

## Repository Settings Required (One-Time Manual Step)

- **Pages source**: Set to **"GitHub Actions"** (not branch-based) in `Settings → Pages → Source`
- This is required before the first `deploy-pages` run; the `github-pages` environment is auto-created on first deploy

## Stack Patterns by Variant

**If deploying to a GH Pages subdomain (no custom domain, e.g. `axiom-laboratories.github.io/axiom/`):**
- `site_url: https://axiom-laboratories.github.io/axiom/` (trailing slash required)
- The `offline` plugin must be disabled for this build to avoid `use_directory_urls: false`
- `mkdocs-swagger-ui-tag` `oauth2RedirectUrl` will correctly resolve under `/axiom/` given the trailing slash

**If deploying to a custom domain (e.g. `docs.axiom.sh`):**
- `site_url: https://docs.axiom.sh/` (root domain, no subpath needed)
- Add a `CNAME` file in `docs/docs/` containing the custom domain, or configure via GH Pages settings
- `offline` plugin conditional still applies

**If the API schema changes:**
- Run the local regen script: generates `openapi.json` by importing the FastAPI app with dummy env vars
- Commit the updated `docs/docs/api-reference/openapi.json`
- The next push to `main` deploys the updated API reference automatically

## Version Compatibility

| Package/Action | Compatible With | Notes |
|----------------|-----------------|-------|
| `mkdocs-material==9.7.5` | `actions/upload-pages-artifact@v3` + `actions/deploy-pages@v4` | No conflicts; material 9.7.x is the current stable maintenance branch (maintenance mode announced Nov 2025 — security fixes only, no breaking changes) |

| `actions/upload-pages-artifact@v3` | `actions/deploy-pages@v4` | Matched pair; v3+v4 required after Jan 2025 deprecation of v2 artifact actions |

| `offline` plugin `!ENV [OFFLINE, false]` | `mkdocs-material==9.7.5` | `!ENV` YAML tag is a standard MkDocs feature, not Material-specific; supported in all MkDocs 1.x versions |
