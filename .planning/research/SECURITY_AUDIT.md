# Security Audit: Production Reliability Milestone

**Date:** 2026-03-05
**Scope:** Phases 1-5 (Output Capture, Retry Policy, Execution History, Environment Tags, Job Dependencies)

## Audit Objectives
1.  **Secret Leakage**: Ensure `ExecutionRecord` output logs do not contain unmasked secrets.
2.  **Environment Isolation**: Verify that `env:` tags strictly prevent cross-environment job assignment.
3.  **Dependency Manipulation**: Check if job dependencies can be used to bypass security checks or cause DoS.
4.  **RBAC Enforcement**: Confirm that new endpoints (History, Node Tagging) require appropriate permissions.
5.  **Injection Risks**: Audit use of `LIKE` queries and JSON deserialization in dependency unblocking.

## Findings

### 1. Secret Leakage & Masking
*Status: VULNERABILITY IDENTIFIED (Medium)*
- **Observation**: `JobService.report_result` takes the raw `output_log` from the node and writes it directly to the `ExecutionRecord` table.
- **Risk**: If a job script prints secrets (e.g., `echo $SECRET_KEY`), those secrets are stored in plain text in the database and visible to anyone with "viewer" access to the history dashboard.
- **Root Cause**: No log scrubbing is performed server-side before persistence.
- **Action**: Implement log scrubbing in `report_result`. A potential fix is to fetch the job payload, decrypt secrets, and use them as patterns to redact from the log lines before saving.

### 2. Environment Isolation (Strict Tags)
*Status: VULNERABILITY IDENTIFIED (High)*
- **Observation**: Nodes report their own `tags` via heartbeat, which are stored in the `Node.tags` column.
- **Risk**: A malicious or misconfigured node can report `env:prod` in its heartbeat. If an operator has not explicitly set `operator_tags` for that node, the system will accept the node's self-reported tags as "effective".
- **Impact**: A "dev" node can steal "prod" jobs by simply claiming to be in the prod environment. This completely bypasses the intended strict isolation.
- **Root Cause**: `JobService._get_effective_tags` falls back to self-reported `tags`.
- **Action**: Change `env:` tag logic. Self-reported tags should NEVER be allowed to contain the `env:` prefix. The server must strip any `env:` tags from heartbeats, ensuring only `operator_tags` can control environment segmentation.

### 3. Dependency Orchestration
*Status: VULNERABILITY IDENTIFIED (Low/Medium)*
- **Observation**: `JobService._cancel_dependents` uses recursion to propagate terminal failures through the dependency graph.
- **Risk**: A deep chain of jobs (e.g., 1000+ jobs in a linear A->B->C... chain) could cause a stack overflow or database transaction timeout during recursive cancellation.
- **Observation**: Searching for dependents uses `LIKE` on a JSON text column.
- **Risk**: Performance bottleneck. While not a direct security breach, it allows an authorized user to create a "noisy neighbor" effect by submitting thousands of BLOCKED jobs, slowing down every `report_result` call.
- **Action**: Implement a depth limit for dependencies or convert recursive cancellation to an iterative approach. For performance, consider a dedicated `job_dependencies` junction table.

### 4. API Authorization
*Status: Minor Issue Identified*
- **Observation**: `/api/executions` and `/api/executions/{id}` only require `Depends(get_current_user)`.
- **Risk**: Any user with a valid account (including those with limited roles) can view the full execution history and all captured logs. If logs contain sensitive data (see Finding #1), this is a significant information disclosure risk.
- **Observation**: `PATCH /nodes/{node_id}` correctly requires `require_permission("nodes:write")`.
- **Action**: Elevate history viewing permissions to `require_permission("history:read")` if that permission exists, or at least `operator` role. Ensure Finding #1 (Log Scrubbing) is implemented to mitigate risk for authorized viewers.
