    Foundry Troubleshooting

    Foundry builds custom Docker images for Axiom Nodes from blueprints and templates. When a build fails, the error appears in the Templates view status column and in the orchestrator logs — this runbook maps each observable symptom to its root cause and recovery steps.

    See Foundry feature guide for background on blueprints, templates, and the Smelter.

    Build Failures

    Template build status shows FAILED

    The Docker build process returned a non-zero exit code — the most common causes are a pip package that does not exist, a base image that could not be pulled, or a Dockerfile syntax error generated by the template engine. The orchestrator captures the last 250 characters of the build output in the template status field.

    Recovery:

    1. Check the orchestrator logs for ❌ Build Failed: followed by the truncated Docker output.
    2. Identify the failure type:
      • "no such package" or "Could not find a version" — the package name in the blueprint is incorrect or does not exist on PyPI. Check the package name and version specifier.
      • "image not found" or "pull access denied" — the base image specified in the blueprint cannot be pulled. Verify the base image name and that your registry is reachable.
      • Syntax error or unexpected content — the blueprint JSON has a malformed entry. Review the blueprint in Foundry → Blueprints and correct any invalid fields.
    3. Fix the blueprint and navigate to Foundry → Templates → [template] → Rebuild to trigger a fresh build.

    Verify: The template status column changes from FAILED to ACTIVE.

    If the build continues to fail after fixing the blueprint, check orchestrator logs for the full Docker build output (docker compose logs agent | grep -A 20 "Build Failed").

    See Foundry → Blueprints for the correct blueprint structure.

    Template build rejected with 403 — unapproved ingredients

    The Smelter is running in STRICT enforcement mode and the blueprint contains one or more packages that are not present in the approved ingredient list. The response body in the orchestrator logs is:

    Build rejected: Blueprint contains unapproved ingredients: [<pkg>, ...]

    The build is rejected before Docker is invoked — no image is produced.

    Recovery:

    1. Check the orchestrator logs for the rejected package names.
    2. Either:
      • Submit an approval request for the missing packages via Foundry → Smelter → Add Ingredient, then rebuild once approved.
      • Remove the unapproved package from the blueprint if it is not required.
    3. If you need to unblock quickly and your deployment policy permits it, switch the Smelter to WARNING mode — the build will proceed and the template will be marked non-compliant rather than rejected.

    Verify: Template builds successfully and the status column shows ACTIVE (or STAGING if Smelt-Check runs).

    Warning

    In WARNING mode, templates with unapproved ingredients are marked is_compliant=False and display an amber badge. They can be deployed, but they do not meet the approval policy. Switch back to STRICT mode after resolving ingredient approvals.

    See Foundry → Smelter for enforcement mode configuration.

    Template build rejected with 403 — ingredient not yet mirrored

    The package is approved in the Smelter but has not yet been synchronised to the local PyPI or APT mirror — the mirror_status is not MIRRORED. In air-gapped environments where node containers cannot reach the public internet, Foundry requires all packages to be available on the local mirror before building. The response body is:

    Build rejected: Ingredient '<pkg_name>' is approved but not yet mirrored (Status: <status>). Wait for mirroring to complete or upload the package manually.

    Recovery:

    1. Wait for the mirror synchronisation job to complete. Mirror status is visible in Foundry → Smelter → Ingredients.
    2. If the sync is taking too long or has stalled, use the Smelter manual upload endpoint to push the package directly.
    3. Once the ingredient status changes to MIRRORED, trigger a rebuild.

    Verify: Template builds successfully with no 403 response.

    See Air-Gap Operation for mirror configuration and offline package upload procedures.

    Build log shows warning: environment_service not found

    The Foundry service builds the Docker context by copying puppets/environment_service/ from the orchestrator container's filesystem into a temporary build directory. If the path is not found — due to a misconfigured volume mount or an incorrect working directory — the generated Dockerfile's COPY step will fail. The warning appears in orchestrator logs before the build starts:

    ⚠️  environment_service not found at <path> — COPY may fail

    Recovery:

    1. Confirm the orchestrator container has access to the puppets source directory. Run docker exec puppeteer-agent-1 ls /app/puppets/environment_service to check whether the path exists inside the container.
    2. If the path is missing, verify the volume mounts in compose.server.yaml include the puppets/ directory.
    3. In production deployments built from the standard Docker Compose setup, environment_service/ is baked into the image — no volume mount is needed. If you are running a custom deployment, verify the path matches what the Foundry service expects.

    Verify: Run the build again and confirm the ⚠️ environment_service not found warning does not appear.

    Tip

    This warning almost always indicates a non-standard deployment path. Standard Docker Compose deployments using compose.server.yaml do not encounter it because environment_service/ is included in the agent image build context.

    Smelt-Check Failures

    Template status is STAGING after a successful build

    The Smelt-Check smoke test (python --version && pip --version) ran inside the newly built image and returned a non-zero exit code — the image was built successfully but the Python environment inside it is not functional. The build pipeline moves the template to STAGING status and logs:

    ❌ Smelt-Check FAILED for <template_name>

    STAGING templates cannot be deployed to nodes.

    Recovery:

    1. Check the orchestrator logs for the Smelt-Check failure line to confirm the template name.
    2. The most common cause is a base image and OS family mismatch. Alpine images use apk and have different package paths than Debian images. Verify that the base_os field in the blueprint matches the OS family of the base image (e.g., alpine in the image name → OS family should be ALPINE).
    3. Fix the blueprint's base image or OS family setting and trigger a rebuild from Foundry → Templates → [template] → Rebuild.

    Verify: Template status changes from STAGING to ACTIVE after the rebuild.

    If the Smelt-Check continues to fail after fixing the OS family, inspect the build output manually: docker run --rm <image_name> python --version to test the built image locally.

    See Foundry → Image Lifecycle for a description of template statuses.

    Template shows amber warning badge in Templates view

    The Smelter is in WARNING mode and detected unapproved or unmirrored ingredients during the build. The build succeeded and the Smelt-Check passed, but the template is marked is_compliant=False. The amber badge is informational — the template can be deployed, but it does not meet the current approval policy.

    Recovery:

    1. This is expected behaviour in WARNING mode. No action is required to deploy the template.
    2. If compliance is required: navigate to Foundry → Smelter, approve the flagged ingredients, switch to STRICT mode for future builds, then rebuild the template.
    3. After rebuilding in STRICT mode with all ingredients approved, the amber badge is removed.

    Verify: After rebuilding with STRICT mode and approved ingredients, the amber badge is absent.

    See Foundry → Smelter for enforcement mode details.

    Registry Issues

    Template shows a last built time but image is not available on nodes

    The Docker build succeeded but the subsequent docker push to the local registry (localhost:5000) failed. The template's status field begins with PUSH_FAILED:. The orchestrator logs the push failure and continues — it does not raise an error or mark the template FAILED — so the dashboard may show a last_built_at timestamp even though the image was never pushed.

    Recovery:

    1. Check orchestrator logs for PUSH_FAILED following the build completion line.
    2. Verify the local registry container is running: docker ps | grep registry. If it is not listed, start it with docker compose -f compose.server.yaml up -d registry.
    3. Once the registry is confirmed running, navigate to Foundry → Templates → [template] → Rebuild to re-trigger the build and push.

    Verify: Check orchestrator logs — the push step should complete without PUSH_FAILED. Confirm the image is in the registry: curl http://localhost:5000/v2/<image_name>/tags/list.

    Warning

    The dashboard last built at timestamp reflects when the Docker build completed, not when the push succeeded. Always check orchestrator logs if nodes cannot pull the image, even if the dashboard shows a recent build time.

    Node container fails to pull image from the local registry at startup

    The node container cannot reach localhost:5000 from inside the Docker bridge network. localhost inside a container resolves to the container itself — not the Docker host. The node logs a pull error at startup.

    Recovery:

    1. Change the image URL in the node's compose file from localhost:5000/<image> to host.docker.internal:5000/<image> or 172.17.0.1:5000/<image>.
    2. Add extra_hosts to the node service definition to ensure hostname resolution: 
      extra_hosts:
  - "host.docker.internal:172.17.0.1"
    3. Restart the node container: docker compose up -d <node_service>.

    Verify: Node container pulls the image successfully and progresses past the pull step in docker logs <node_container>.

    Tip

    The validation node compose files in mop_validation/local_nodes/ use the correct networking pattern. Copy the extra_hosts block and image URL format from there if setting up a new node.
