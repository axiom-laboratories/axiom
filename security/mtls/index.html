    mTLS & Certificates

    Mutual TLS (mTLS) is a security protocol where both parties in a connection — the orchestrator and each Axiom Node — present certificates to prove their identity. Unlike standard TLS, where only the server is verified, mTLS also verifies the client. This means the orchestrator will refuse connections from any node that does not hold a certificate signed by its own Root CA.

    PKI Terminology

    CA (Certificate Authority) — A trusted entity that issues and signs certificates. Axiom operates its own Root CA, making it the sole authority for trusting node identities.

    CSR (Certificate Signing Request) — A document generated by a node that contains its public key and identity information. The node sends the CSR to the CA, which signs it to produce a certificate.

    Certificate — A digitally signed document binding a public key to an identity (hostname). Nodes present their certificate on every connection to prove they are who they claim to be.

    Certificate serial number — A unique integer assigned to each certificate by the CA. Used to identify the certificate in revocation records.

    CRL (Certificate Revocation List) — A signed list of certificate serial numbers that have been revoked before their expiry date. Revoked certificates are treated as invalid even if they have not expired.

    Enrollment — The process by which a new node obtains its first signed certificate. The node generates a CSR, submits it to the orchestrator, and receives a signed certificate in return.

    How Axiom Uses mTLS

    • The orchestrator operates a Root CA that signs all node certificates
    • Nodes enroll by sending a CSR and receiving a signed client certificate
    • Every /work/pull and /heartbeat call requires the node to present its client certificate
    • Revoked certificates are tracked in a CRL served at GET /system/crl.pem
    • Node-facing endpoints (/api/enroll, /work/pull, /heartbeat) use mTLS instead of JWT-based auth — this is why they appear unauthenticated in the API reference

    The Root CA

    The Root CA is the trust anchor for the entire fleet. It is created automatically when the orchestrator starts for the first time, if no CA files already exist in secrets/ca/.

    CA parameters:

    Parameter Value
    Key algorithm RSA 4096-bit
    Signature algorithm SHA-256
    Validity 10 years (3650 days)
    Subject CN Axiom Root CA

    CA file locations:

    File Contents
    secrets/ca/root.key Root CA private key
    secrets/ca/root.crt Root CA certificate (public)

    Protect the Root CA private key

    secrets/ca/root.key is the most sensitive file in the deployment. Anyone with access to this file can sign arbitrary certificates and enroll rogue nodes without going through the orchestrator's enrollment API. Back it up to offline storage and restrict filesystem access to the orchestrator process only.

    Advanced: cert-manager integration

    For Kubernetes deployments using cert-manager, the Root CA can be loaded from /app/global_certs/root_ca.crt instead of the self-managed secrets/ca/ path. The Docker Compose deployment described in this guide uses the self-managed CA path.

    The JOIN_TOKEN

    The JOIN_TOKEN is a base64-encoded JSON object containing the Root CA certificate. Nodes use it to establish initial trust with the orchestrator — without it, a node has no way to verify it is talking to the correct orchestrator and not an impersonator.

    Generate a JOIN_TOKEN:

    1. Navigate to Admin → System
    2. Click Generate Join Token
    3. Copy the token — it contains the Root CA certificate embedded in base64

    Use the token when starting a node:

    JOIN_TOKEN=<YOUR_JOIN_TOKEN> docker compose -f node-compose.yaml up -d

    Who should receive the JOIN_TOKEN?

    The JOIN_TOKEN contains the public Root CA certificate — it is not itself a cryptographic secret. However, any node that receives a valid JOIN_TOKEN can attempt to enroll. Do not share it beyond nodes you intend to add to the fleet. If a token is shared accidentally, generate a new one; old tokens remain valid until the Root CA itself changes, so rely on the enrollment audit log to detect unexpected enrollments.

    Node Enrollment

    Enrollment is fully automatic when a node container starts with a valid JOIN_TOKEN. The steps below describe what happens under the hood:

    1. The node reads JOIN_TOKEN from its environment and decodes it to extract the Root CA certificate
    2. It uses the Root CA certificate to establish a trusted TLS connection to the orchestrator (verifying that the server certificate chains up to the same Root CA)
    3. The node generates a new RSA private key and creates a Certificate Signing Request (CSR) with its hostname as the Common Name (CN)
    4. The node sends the CSR to POST /api/enroll on the orchestrator
    5. The orchestrator signs the CSR and returns a client certificate valid for 825 days (~2.25 years)
    6. The node stores its private key and certificate in its secrets/ volume and uses them for all subsequent /work/pull and /heartbeat calls

    After enrollment, the node appears in Nodes with status Active. If enrollment fails — for example because the JOIN_TOKEN is malformed, the orchestrator is unreachable, or the node's certificate files from a previous enrollment are still present — the container exits and logs the error.

    Node identity persistence

    The node's identity (a UUID derived from its certificate CN) is persisted in secrets/. If the container restarts, it scans secrets/ for an existing node-*.crt file and reuses that identity rather than generating a new one. This prevents duplicate node entries in the dashboard after container restarts.

    Certificate Revocation

    Revoking a certificate immediately blocks that node from polling for work or re-enrolling. Use revocation when a node is decommissioned, a certificate is compromised, or a node container is replaced with a new identity.

    Steps to revoke a node:

    1. Navigate to Nodes in the dashboard
    2. Find the node and click Revoke
    3. Confirm the revocation when prompted

    What revocation does:

    • The node's status is set to OFFLINE
    • The certificate serial number is added to the CRL
    • All subsequent /work/pull and /api/enroll requests from that node return HTTP 403

    The CRL is available at GET /system/crl.pem (unauthenticated, so external tooling can fetch it) and has a next_update interval of 7 days.

    Revocation is permanent

    Revoking a certificate is permanent. The node cannot re-enroll using the same certificate. If the node container is still running, it will immediately start receiving HTTP 403 errors on its next /work/pull poll. Stop the container before revoking unless you intend to use the 403 response as a signal.

    Certificate Rotation

    Node certificates are valid for 825 days (~2.25 years). Before a certificate expires, rotate it by re-enrolling the node with a new certificate. Rotation requires temporarily stopping the node.

    Prerequisites — confirm before starting:

    • [ ] You have access to stop and restart the node container
    • [ ] You have orchestrator admin credentials to generate a new JOIN_TOKEN
    • [ ] The node is not currently running a critical job (check the Jobs view and confirm no job is running on this node)

    Rotation procedure:

    Step 1 — Generate a new JOIN_TOKEN. In the dashboard, navigate to Admin → System → Generate Join Token. Copy the new token.

    Step 2 — Stop the node container.

    docker compose -f node-compose.yaml down

    Step 3 — Delete the node's certificate files.

    rm -f secrets/node-*.crt secrets/node-*.key

    Why delete the cert files?

    This forces the node to generate a new identity on next start. The node ID is derived from the certificate CN — deleting the old cert means the node will enroll fresh with a new ID. The orchestrator will register it as a new enrollment rather than trying to reuse the old (soon-to-be-revoked) cert.

    Step 4 — Start the node with the new JOIN_TOKEN.

    JOIN_TOKEN=<NEW_JOIN_TOKEN> docker compose -f node-compose.yaml up -d

    Step 5 — Verify the node has enrolled. In the dashboard, navigate to Nodes and confirm the node appears with status Active. The last-seen timestamp should be within the last minute.

    Point of no return — do not proceed until the new node is Active

    Do not proceed to Step 6 until you have confirmed the node is Active in the dashboard. After Step 6, the old certificate is permanently revoked. If the new enrollment has not succeeded, the node will be unable to re-enroll and you will need to investigate the enrollment failure before any work can be dispatched to it.

    Step 6 — Revoke the old certificate. In the dashboard, find the previous node entry (it may appear as Offline) and click Revoke. Confirm the revocation.

    Step 7 — Verify the CRL.

    curl -s https://<YOUR_HOST>/system/crl.pem

    The response should be a PEM-encoded CRL. Confirm the old node no longer appears as Active in the dashboard.

    If the node appears as a separate entry after rotation

    If the old and new node entries have different names or UUIDs, it is because the old and new certificates have different hostnames or CNs. This is expected. Delete the old offline entry from the Nodes view after confirming the new node is active and working.
