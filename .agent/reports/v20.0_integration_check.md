### Data Flow Lineage

```
Dashboard Form
  ├─ memory_limit input → JobCreate.memory_limit (Phase 120 validators)
  ├─ cpu_limit input → JobCreate.cpu_limit (Phase 120 validators)
  └─ Submit → POST /api/jobs

POST /api/jobs Handler (main.py)
  ├─ Receives JobCreate with limits
  └─ Calls job_service.create_job()

job_service.create_job() (Phase 121)
  ├─ Parses memory_limit with parse_bytes()
  ├─ Compares against node.job_memory_limit (per-node limit)
  ├─ Rejects with 422 if insufficient capacity
  ├─ Stores in Job table (Phase 120 db schema)
  └─ Returns JobResponse

Node polls POST /work/pull
  └─ Calls job_service.pull_work()

job_service.pull_work() (Phase 120-121)
  ├─ Queries Job from database
  ├─ Populates WorkResponse.memory_limit (Phase 120)
  ├─ Populates WorkResponse.cpu_limit (Phase 120)
  └─ Returns serialized JSON to node

node.py poll_for_work()
  └─ Parses WorkResponse JSON

node.py execute_task() (Phase 122)
  ├─ Extracts memory_limit from job dict
  ├─ Validates with parse_bytes()
  ├─ Extracts cpu_limit from job dict
  ├─ Validates with parse_cpu()
  ├─ Reports structured error if invalid
  └─ Calls runtime.run(memory_limit, cpu_limit)

Container Runtime
  ├─ Receives --memory flag
  ├─ Receives --cpus flag
  └─ Enforces resource constraints
```

## E2E Flow Verification

### Flow 1: Operator Dispatches Job with Memory Limit
**User Action**: Enter memory_limit="512m" in job dispatch form
**Path**: Dashboard → API → DB → Node → Runtime
**Result**: PASS ✓
- Form accepts "512m" (regex validator)
- API validates format (Phase 120)
- Admission check parses and compares (Phase 121)
- Job stored in DB with limit
- WorkResponse populated and sent to node
- Node validates and applies limit

### Flow 2: Operator Creates Scheduled Job with Limits
**User Action**: Create ScheduledJob with memory_limit="1g"
**Path**: Dashboard → DB → Scheduler → Job → Node → Runtime
**Result**: PASS ✓
- Form accepts "1g"
- ScheduledJob stored with limit (Phase 121)
- At cron fire: scheduler_service.fire() passes limit to create_job()
- create_job() admission check validates
- Job created with inherited limit
- Node executes with limit

### Flow 3: Operator Checks Dispatch Diagnosis
**User Action**: Click "Diagnosis" on PENDING job
**Path**: Frontend → API → Database → Node Capacity Calculation
**Result**: PASS ✓
- Frontend calls GET /jobs/{guid}/dispatch-diagnosis
- get_dispatch_diagnosis() returns reason + nodes_breakdown
- Frontend displays table: Node ID | Capacity | Used | Available | Fits?
- Operator sees why job hasn't dispatched

### Flow 4: Node Receives Invalid Memory Format
**Scenario**: Orchestrator sends memory_limit="10x" (invalid)
**Path**: WorkResponse → Node Validation → Error Report
**Result**: PASS ✓
- Node extracts memory_limit="10x"
- parse_bytes("10x") raises ValueError
- execute_task() catches and returns structured error:
  - error: "Invalid memory_limit format"
  - value: "10x"
  - expected: "e.g. 512m, 1g, 2Gi"
- Job marked FAILED
- logger.warning() records event

### Flow 5: Node Receives Invalid CPU Format
**Scenario**: Orchestrator sends cpu_limit="invalid" (not float)
**Path**: WorkResponse → Node Validation → Error Report
**Result**: PASS ✓
- Node extracts cpu_limit="invalid"
- parse_cpu("invalid") raises ValueError
- execute_task() catches and returns structured error:
  - error: "Invalid cpu_limit format"
  - value: "invalid"
  - expected: "e.g. 2, 0.5, 1.0"
- Job marked FAILED
- logger.warning() records event
